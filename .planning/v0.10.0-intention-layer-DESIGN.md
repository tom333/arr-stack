# Design — v0.10.0 : Couche d'intention (méta-orchestrateur, P2 incrémentale)

**Date :** 2026-05-31
**Statut :** Validé (brainstorming) — input pour `/gsd-new-milestone` v0.10.0
**Auteur :** Thomas + Claude (session brainstorming)

---

## 1. Motivation

Le système tourne en prod (`:0.15.0`, 416+ tests) — mais 3 insatisfactions structurelles, **une seule cause racine** :

> Le YAML est la source de vérité, et l'UI ne fait que le *refléter* (1:1) — elle ne l'*abstrait* pas.

1. **UI peu utile** — ~5 020 LOC (FastAPI + Svelte) qui rendent le schéma champ-par-champ. Miroir du YAML, abstraction nulle. Coût élevé, valeur faible.
2. **YAML trop gros** — arrconf.yml 316 + configarr.yml 459 + values.yaml 617 = ~1 400 lignes hand-maintained. Verbosité = CF FR dupliqués + expansion `categories[]` + verbosité configarr/recyclarr.
3. **Pas dynamique** — `categories[]` EST déjà déclaratif, mais le couplage downstream (filesystem, profils) s'arrête au bord du modèle déclaratif. Ajouter une catégorie n'est pas trivial.

**Prototype existant de la solution :** `categories[]` = **10 lignes d'intention → 6 reconcilers**. Le refactor n'invente rien : il **généralise** ce pattern.

---

## 2. Décision : P2 — couche d'intention incrémentale

Choix parmi 3 approches :
- **P1** couche d'intention complète d'un coup — rejeté (big-bang sur un prod qui marche).
- **P2** intention incrémentale — **RETENU** (même destination que P1, valeur livrée par tranche, pas de réécriture brutale).
- **P3** statu quo + ajouts YAML hand-edited — rejeté (les 3 douleurs restent).

### Architecture cible

```
intent.yml  ← seule chose éditée à la main (UI plus tard)
   │
   ▼  arrconf generate   (fonction pure : intent → configs verbeuses)
arrconf.yml · configarr.yml · qbit_manage/config.yml · cross-seed/config.js   ← GÉNÉRÉS (read-only, committés)
   │
   ▼  arrconf apply  +  configarr   (reconcile — INCHANGÉ)
APIs *arr/qbit/jellyfin  +  fichiers montés
```

### Principe de frontière (nouveau — à acter en ADR)

- **Absorber** (générer la config) : tout ce qui expose un fichier déclaratif ou une API propre → sagas, cross-seed, qbit_manage, + tout l'existant.
- **Déployer seulement** (alias Helm, config à la main/UI) : DB/UI-only → autobrr, Tdarr/FileFlows, cleanuparr (déjà ainsi).

### Modèle de génération : G1 (local + committé)

`arrconf generate` tourne **en local** ; les YAML/JS générés sont **committés**. CI = garde-fou d'idempotence (`arrconf generate && git diff --exit-code`).

- ✅ Colle à la discipline ADR-6 (snapshots), diff visible en PR, aligné co-bump pin.
- Rejeté : G2 (in-cluster, perd le diff Git/ADR-6) ; G3 (commits auto, bruit + interaction auto-tagger).

Le générateur **réutilise le pattern `arrconf/generators/categories.py`** (générateurs purs déjà en place) — extension, pas réinvention.

## 5. ADR

- **ADR-1** (custom Python vs IaC) : **tient**. awesome-arr = aucun provider Terraform/Pulumi ; Buildarr (le plus proche) stagnant + sans `categories[]`/qBit/Jellyfin → migrer = downgrade. (devopsarr/Terraform pour la moitié *arr-CRUD = Phase 28 séparée, optionnelle.)
- **ADR-5** (frontière arrconf/configarr) : **tient + étendu**. La couche intention se place *au-dessus* d'arrconf ET configarr ; configarr reste seul à apply TRaSH.
- **ADR-nouveau (à rédiger)** : couche d'intention + frontière *absorber vs déployer*.
