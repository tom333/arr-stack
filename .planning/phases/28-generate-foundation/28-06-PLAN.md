Then a short intro paragraph, then bolded sub-points covering the THREE mandatory content items (D-12 / 28-PATTERNS.md 607-613), sourced from DESIGN §2 + §5:

- **Contexte / Layer:** the intention layer sits ABOVE both arrconf and configarr. Flow: `intent.yml` (seul fichier hand-edited) → `arrconf generate` (fonction pure) → configs verbeuses **committées** → `arrconf apply` / configarr reconcilient (inchangés). Generalizes the `categories[]` pattern (1 ligne d'intention → N reconcilers) into an explicit layer.
- **Décision — frontière absorber vs déployer-seulement:** "Absorber" = tout outil ayant un fichier/API de config déclaratif (cross-seed, qbit_manage) → sa config est **générée** par la couche d'intention depuis `intent.yml`. "Déployer-seulement" = outils config DB-only / UI-only (autobrr, cleanuparr) → déployés comme alias Helm uniquement, **aucune** config intention-générée.
- **Extension d'ADR-5:** configarr reste le **seul appliqueur TRaSH**. La couche d'intention ne touche JAMAIS `quality_profiles` / `custom_formats` / `quality_definitions` / `media_naming` — frontière dure inchangée. La couche d'intention se place au-dessus d'ADR-5, ne le contredit pas.
- **Conséquences:** G1 (générer en local + committer) préserve le diff Git + la discipline ADR-6 (review-in-PR). `generate` et `apply` restent découplés (D-06). Le drift est bloqué par la garde CI d'idempotence (INTENT-03).
- **Alternatives rejetées:** G2 (génération in-cluster — perd le diff Git), G3 (auto-commit — bruit Git + interaction auto-tagger), P1 (big-bang couche complète — réécriture brutale d'un prod qui marche). Voir REQUIREMENTS.md Out of Scope.
