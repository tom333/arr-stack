<decisions>
## Implementation Decisions

### Trigger fix mechanism
- **D-05.1-TRIGGER-01** : Utiliser `repository_dispatch` (Option C) plutôt que PAT (A) ou GitHub App token (B).
  - **Why** : Pas de secret-à-rotater, pas de dépendance à un compte utilisateur, tout reste in-repo et auditable via les workflow files. Les `repository_dispatch` events ne sont PAS soumis à la politique anti-loop GitHub (seul `on: push:` est filtré quand le push vient de `GITHUB_TOKEN`).
  - **Rejets** :
    - **A (PAT)** rejeté : rotation manuelle, scope `contents: write` + `workflow: write` excessif pour le seul besoin de fire un workflow, single-point-of-failure si le propriétaire perd l'accès.
    - **B (GitHub App)** rejeté : setup disproportionné pour un repo solo, complexité opérationnelle non justifiée.
  - **How to apply** : Le job `tag` de `chart-lint.yml` gagne un step `gh api -X POST repos/${{ github.repository }}/dispatches -f event_type=arrconf-image-build -f client_payload[tag]=${{ steps.tag.outputs.new_tag }}` (ou équivalent `actions/github-script`). `arrconf-image.yml` gagne `on: repository_dispatch: types: [arrconf-image-build]`.

### Scope placement
- **D-05.1-SCOPE-01** : Mini-phase Phase 5.1 (insertion décimale) plutôt que Plan 05-09 sous Phase 5.
  - **Why** : Le scope (CI + workflow plumbing) ne touche aucun code/values/chart de Phase 5 (qBit reconciler + ADR-7 split). Le sujet est orthogonal. Suivre le pattern Phase 2.1 / 2.2 (insertions décimales pour fix architectural urgent) maintient une roadmap lisible et n'altère pas le SUMMARY/VERIFICATION de Phase 5.
  - **How to apply** : Phases décimales = insertions urgentes (cf. ROADMAP.md ligne 13). 5.1 BLOQUE Plan 05-08 (reste pending) mais ne réécrit pas Phase 5.

### Orphan tag policy
- **D-05.1-ORPHAN-01** : v0.3.0 est accepté comme tag-orphelin. Pas de retag, pas de rescue manuel `workflow_dispatch`.
  - **Why** : Réécrire l'historique des tags (force-push, retag) viole CLAUDE.md (« Ne pas amender un tag de release publié »). Un rescue `workflow_dispatch` ponctuel polluerait l'historique des runs (le sha ref v0.3.0 produirait `:0.3.0` postérieurement, semantically inconsistent avec le tag créé en 2026-05-15 mais image build en 2026-05-15+N). Le coût d'un tag sans image semver = NUL (le code Phase-5 reste accessible via `:sha-ef7681a` et le prochain tag `:0.3.1` fournira la première image semver propre).
  - **How to apply** : Documenter dans le SUMMARY de Phase 5.1 que v0.3.0 est un tag-orphelin (date, sha, raison). Aucune action de cleanup. Plan 05-08, quand il reprendra, ciblera `:0.3.1` (ou la version qui aura été bumpée par Renovate dans `values.yaml`).

### Defense in depth (workflow_dispatch rescue path)
- **D-05.1-RESCUE-01** : `workflow_dispatch` ajouté à `arrconf-image.yml` comme filet de sécurité futur (pas pour rescue v0.3.0).
  - **Why** : Le pattern actuel n'a aucun chemin manuel pour rebuild une image si `repository_dispatch` lui-même échoue (rate-limit, API outage, payload mal formé). Un `workflow_dispatch` avec input `tag_ref` permet à l'opérateur de fire le build manuellement depuis l'UI Actions sans toucher au workflow file.
  - **How to apply** : Input optionnel `tag_ref` (string, default: `main`). Le step de checkout utilise `ref: ${{ github.event.inputs.tag_ref || github.event.client_payload.tag || github.ref }}`. Documenter dans SUMMARY que c'est rescue-only, NE PAS devenir le chemin nominal.

### Legacy push-tags trigger
- **D-05.1-LEGACY-01** : Conserver `on: push: tags: ['v*']` dans `arrconf-image.yml`.
  - **Why** : Filet pour les tags poussés manuellement via local PAT (cas hors-CI : opérateur push un tag depuis sa machine pour debugging). Coût additionnel ~0, suppression du legacy = perte de redondance gratuite.
  - **How to apply** : Garder tel quel, ajouter `repository_dispatch` et `workflow_dispatch` à côté, pas en remplacement.

### CI regression guard
- **D-05.1-GUARD-01** : Garde CI déférée à Q-05.1-1 (open question).
  - **Why** : Un guard "fail si `repository_dispatch:` absent de arrconf-image.yml" est facile à écrire mais ajoute du noise dans chart-lint.yml. Le risque de revert silencieux est faible (workflows touchés rarement, PRs reviewées). Décision tranchée en plan-phase si besoin.
  - **How to apply** : Si Q-05.1-1 = OUI → step dans `chart-lint.yml::lint` : `grep -q "repository_dispatch:" .github/workflows/arrconf-image.yml || exit 1`. Si NON → documenter le risque dans SUMMARY et clore.

### values.yaml bump path
- **D-05.1-BUMP-01** : Laisser arr-stack Renovate auto-bumper `arrconf.tag` une fois `:0.3.1` publiée.
  - **Why** : Le customManager dans `renovate.json:7-15` est confirmé en place et automerge les minor/patch (`renovate.json:17-23`). Inclure le bump dans la PR Phase 5.1 nécessite de référencer un tag qui n'existe pas encore (`:0.3.1` se crée seulement après merge de Phase 5.1). Coupler le bump à la PR force soit (a) deux PRs séquentielles ("merge fix, attendre 5min, ouvrir bump PR"), soit (b) un tag manuel pre-merge — les deux dégradent l'UX. Renovate handle ça depuis Phase 4.
  - **How to apply** : Aucune modif de values.yaml dans Phase 5.1. Après merge, attendre la PR Renovate auto-générée (latence ≤ 1 h per renovate scheduling) puis auto-merge.

</decisions>
