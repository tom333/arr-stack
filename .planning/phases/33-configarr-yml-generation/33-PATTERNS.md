**Determinism helper** (lines 193-203 of intent.py):
```python
def _sort_dict(d: Any) -> Any:
    """Recursively sort dict keys for deterministic YAML output (SC#4)."""
    if isinstance(d, dict):
        return {k: _sort_dict(v) for k, v in sorted(d.items())}
    if isinstance(d, list):
        return [_sort_dict(x) for x in d]
    return d
```
`_sort_dict` is NOT exported from `intent.py` ŌĆö it must be imported explicitly (`from arrconf.generators.intent import _sort_dict`) or duplicated. Importing a private symbol is acceptable here since both files are internal.

**Core generator pattern** (lines 206-221 of intent.py):
```python
def generate_arrconf_yml(intent_cfg: IntentConfig) -> str:
    """Pure: IntentConfig -> arrconf.yml content (apps pass-through)."""
    from ruyaml import YAML

    yaml = YAML(typ="safe")
    yaml.default_flow_style = False
    stream = io.StringIO()
    yaml.dump(_sort_dict(dict(intent_cfg.apps)), stream)
    return _ARRCONF_HEADER + stream.getvalue()
```
`generate_configarr_yml` mirrors this: `YAML(typ="safe")`, `io.StringIO`, `_sort_dict`, header prepended. Difference: the function builds the output dict via profile routing and pass-through merge before calling `yaml.dump`.

**String-construction pattern for `!env` tags** (lines 94-117 of intent.py ŌĆö `generate_qbit_manage`):
```python
lines: list[str] = [_QBM_HEADER]
lines += [
    "qbt:",
    f"  host: {cfg.qbt_host}",
    "  user: !ENV QBT_USER",   # ŌåÉ !ENV emitted as string literal, not a YAML tag
    "  pass: !ENV QBT_PASS",
    "",
]
# ...
return "\n".join(lines)
```
For the `api_key: !env SONARR_API_KEY` lines in the pass-through skeleton, use the same string-construction approach: either emit those lines via `"\n".join(lines)` with literal `!env VAR` strings, OR store `api_key` as a quoted string `"!env SONARR_API_KEY"` in `intent.yml` and add a custom representer to ruyaml that converts the string pattern back to a YAML tag at emit time.

The **recommended approach** (consistent with `generate_qbit_manage` precedent): keep the pass-through section as a `dict[str, Any]` in `IntentConfig.configarr`, but post-process the serialized string to replace quoted `"!env VAR"` values with unquoted `!env VAR` tag syntax. Alternatively emit the entire pass-through block from a pre-serialized string stored in `intent.yml`. The planner must pick one and document it.

def _load_configarr() -> dict[str, Any]:

## Key Constraints for Planner

### ADR-5 cross-package boundary (HARD)
`tools/arrconf-ui/arrconf_ui/configarr_config.py` line 14 states explicitly:
> "ADR-5: This file lives in `tools/arrconf-ui/arrconf_ui/` ONLY ŌĆö NEVER `tools/arrconf/`."

`QualityProfile`, `QualityGroup`, `Upgrade`, `ResetUnmatchedScores`, `CustomFormat`, `AssignScoresTo` MUST be redefined in `intent_config.py` or `generators/configarr.py`. Use minimal typing ŌĆö the generator only needs `custom_formats[].trash_ids`, `custom_formats[].name`, and `custom_formats[].score` for routing; the rest of the QP body can be `dict[str, Any]`.

### Profile rename (D-33-04) ŌĆö planner must decide Option A vs Option B
- **Option A:** Rename `Profile = Literal["general", "anime", "family"]` ŌåÆ `Literal["MULTi.VF", "Anime", "Family"]` in `resources/categories.py`. Updates `intent.yml` category values AND ~100 test fixtures AND 3 production comparisons in `__main__.py:85`, `generators/categories.py:239`, `audit.py:389`.
- **Option B:** Keep enum unchanged. Add mapping `{"general": "MULTi.VF", "family": "Family", "anime": "Anime"}` in `generate_configarr_yml`. `intent.yml` category profile values stay as `general`/`family`/`anime`. Zero test churn.

The CONTEXT.md phrase "category.profile est renomm├® **dans intent.yml**" is consistent with Option B (only the YAML file representation changes, not the enum).

### `_sort_dict` visibility
`_sort_dict` is a private function in `intent.py` (prefixed `_`). The planner must decide: export it (rename to `sort_dict`, add to `__all__`), import the private symbol from `generators.intent`, or duplicate it in `generators/configarr.py`.

### Co-bump rule (CLAUDE.md)
Any commit touching `tools/arrconf/**` must also bump `charts/arr-stack/values.yaml#arrconf.image.tag` in the same commit.
