## Operator Visual Gate FAILED — Plan 06 NOT COMPLETE

**Status:** Plan 02.2-06 is INCOMPLETE. Phase 02.2 is BLOCKED on a real regression discovered by the Task 6.4 operator visual UAT.

**Discovered:** 2026-05-09 (operator manual UI test, post-Plan-06 automated checks)
**Symptom:** Sonarr's "Test" button on the qBittorrent download client returns 401/403 (Auth failed). The automated dispositive checks all PASSED (priority restored 5→1, `put_force_save_used` emitted, no HTTP 400, `manual_nudge_used: NO`), but the human-verify gate detected the credential-side regression that the priority-only diff did not surface.

**Hypothesis (confirmed by forensic):** The v0.1.4 `_ArrV3Client.put()` `?forceSave=true` UPDATE PUT instructed Sonarr to bypass pre-save validation and accept the merged body — which carried the API mask `"********"` as the password value (per Phase 2.1 helper's mask-passthrough behavior, lessons §"Sonarr's API masks privacy=password fields"). Sonarr stored `********` as the literal password in its encrypted store. Sonarr's GET-side serialization then continues returning `"********"` because the masked field is now indistinguishable from a "real" stored credential — making the regression invisible to snapshot diff. When the operator clicks Sonarr's "Test" button, Sonarr decrypts its stored password (`********`) and POSTs it as the literal credential to qBittorrent, which correctly rejects it with 401/403.

This is the architectural risk explicitly accepted by ADR-8 — the trusted-controller bypass is unsafe when the merge helper preserves an API mask token as if it were a real credential value. Phase 2.1's helper protects empty-string values by substituting cluster's stored value, but cluster's stored value IS the API mask, so the protection is circular for `privacy=password` fields.
