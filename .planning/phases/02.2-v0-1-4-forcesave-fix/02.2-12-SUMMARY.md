## Decisions Made

1. **testall endpoint vs body-based test:** Used `POST /api/v3/downloadclient/testall` (not `POST /api/v3/downloadclient/test` with body from GET). The body-based endpoint re-submits the GET response which contains `"password": "********"` — Sonarr would re-submit the mask as the credential and fail. `testall` uses internally stored credentials, equivalent to clicking the UI "Test" button.

2. **Plan 12.2 timing of port-forward failure:** The initial attempt at Task 12.2 body-based test returned HTTP 400 (auth failure), which correctly confirmed the body-based approach fails. This is documented to clarify why the Plan specified `testall` semantics and why snapshot diff cannot detect credential regression.

3. **Drift demo endpoint:** Used port-forward to Sonarr at 8989 throughout (no additional port-forward changes needed). The qBittorrent port-forward at 8085 was started for reconnaissance but not needed for the drift demo (arrconf in-cluster already has access to qbittorrent.selfhost.svc.cluster.local:8080).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Initial Task 12.2 used body-based POST /api/v3/downloadclient/test which always fails when masked fields are present**

- **Found during:** Task 12.2 first attempt
- **Issue:** The plan said `curl -X POST /api/v3/downloadclient/test -d @<(kubectl get ... json)` — but the GET-sourced body contains `"password": "********"`. Posting this body back to `/test` makes Sonarr try to authenticate to qBit with the literal mask string, returning HTTP 400.
- **Fix:** Used `POST /api/v3/downloadclient/testall` (no body) — the correct endpoint that uses internally stored credentials.
- **Verification:** HTTP 200 + isValid:true from testall; body-based endpoint returned HTTP 400 as expected.
- **Impact:** The actual behavioral dispositive is stronger — testall IS the API equivalent of the UI Test button click (same code path in Sonarr). The body-based endpoint is a pre-commit validation path, not the behavioral test.

## Issues Encountered

- **testall vs test endpoint distinction:** The Sonarr API has two test endpoints with different semantics. The body-based `/test` re-validates the submitted body (and will fail with masked fields), while `/testall` tests all configured download clients using their stored credentials. This is the correct behavioral test. Documented as a pattern for future phases.
