<objective>
Documentation-only plan: refine ADR-8 in `spec.md` §11 to reflect the v0.1.5 mitigation strategy. The current ADR-8 (committed at 571d9a1 in Plan 03) explicitly accepted the trusted-controller bypass risk. That risk was realized in production (Plan 06 visual gate failure). v0.1.5 (Plan 08) layers a merge-level omit-by-privacy strategy on top of the existing HTTP-layer bypass. The ADR must capture this without rewriting history.

**Refinement strategy: append ADR-8.1 as a sub-section** (NOT rewrite ADR-8, NOT renumber). This matches CLAUDE.md "Décisions structurantes" append-only convention and ADR-1..7 historical pattern.

The original ADR-8 stays as the architectural decision. ADR-8.1 documents:
1. **What was realized**: the previously-accepted "If Sonarr ever introduces a meaningful pre-save check we'd actually want, arrconf bypasses it" risk concretely manifested as credential overwrite for `privacy=password|userName` fields. The pre-save check Sonarr does for credentials IS meaningful (it re-authenticates against the external service). v0.1.4 bypassed it; v0.1.4 + Phase 2.1 mask passthrough = overwrite.
2. **The fix**: merge-layer omit by `privacy=password|userName` field metadata (Option A from D-02.2-AUTH-REGRESSION required-action #2). Keeps `?forceSave=true` for non-credential drift correction; sidesteps it for credential fields where the body cannot be valid by construction.
3. **The two-layer stance**: ADR-8 (HTTP-layer bypass) + ADR-8.1 (merge-layer omit) = layered defense. The forceSave bypass is correct when the body is correct; the merge-layer omit ensures credential-bearing bodies are correct (by construction — credential entries are absent, Sonarr preserves the stored value).
4. **Scope-out reaffirmed**: qBittorrent (Phase 5) and Jellyfin (Phase 7) are still OUT OF SCOPE for both layers (different PUT semantics; client classes do NOT inherit from `_ArrV3Client`).
5. **Trade-off accepted**: the omit-by-privacy strategy treats `privacy=userName` fields uniformly with `privacy=password` — even when the cluster-side userName value is a real string (e.g. "admin"). The architectural symmetry is intentional: it would be strange to omit password but pass-through username; the credential is the (username, password) pair. arrconf's env-only secret discipline (CLAUDE.md "Variables d'environnement") means YAML never carries either, so the operator-driven UI re-entry covers both.

This plan is parallel-eligible with Plan 08 (different files: `spec.md` vs `differ.py` + tests). Both can run in Wave 7. They share Plan 07 as upstream dependency (Plan 09 references the test contract Plan 07 introduced and Plan 08 makes pass).

Wave 7 — depends on Plan 07 (RED test exists; ADR cites the test contract). Independent of Plan 08 file-modification-wise.
