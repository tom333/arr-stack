## CR-01 Closure Confirmation

**Status: CLOSED**

The guard is correctly implemented at `differ.py:139-152`:

```python
v = des_f.get("value")
if cur_privacy in ("password", "userName"):
    if v == "" or v is None:
        log.info("merge_field_omitted_credential", ...)
        continue
    # Desired has a real value: user intends to update the credential.
    merged_fields.append(des_f)
    continue
```

The omit branch only fires when `v` is empty or None. A non-empty desired credential value
falls through to `merged_fields.append(des_f)` and reaches the PUT body. The docstring at
lines 97-107 accurately reflects the implemented behavior. No silent drop can occur for
non-empty credential values.

The new test at `test_differ.py:387-415` correctly exercises the pass-through path:
`cur` carries `privacy="password"`, `des` carries `value="new_value"`, and the test asserts
both presence in the merged fields dict and the exact value. The logic is sound.

## Warnings

### WR-01: Incomplete credential-privacy enumeration — `apiKey` and `token` not covered

**File:** `tools/arrconf/arrconf/differ.py:140`

**Issue:** The omit-credential guard checks only `cur_privacy in ("password", "userName")`.
Sonarr/Prowlarr also define `"apiKey"` and `"token"` as privacy values (used by indexers and
notification webhooks). The `merge_fields_for_put` docstring at line 96 explicitly claims the
function is "Generic over T: BaseModel so Phase 3 Radarr/Prowlarr reconcilers can reuse it
unchanged (D-33)." But Phase 3 indexer reconcilers will hit `apiKey`/`token` credential fields
and fall through to the empty-value-substitute branch — reintroducing the exact regression that
triggered this hotfix for those field types.

This is a latent bug planted by the D-33 reusability claim combined with an incomplete
enumeration. A future maintainer wiring up the Prowlarr indexer reconciler would not see any
test failure because the existing tests only use `DownloadClient` models.

**Status:** Deferred to Phase 3 per prior review. No change.

**Fix:** Either enumerate all known credential privacy values, or switch to an exclusion-based
approach:

```python
# Approach A: complete enumeration (explicit, auditable)
_CREDENTIAL_PRIVACY_VALUES: frozenset[str] = frozenset(
    {"password", "userName", "apiKey", "token"}
)

# In merge_fields_for_put, replace the inline tuple:
if cur_privacy in _CREDENTIAL_PRIVACY_VALUES:
    ...
```

### WR-02: Misleading docstrings on pre-existing `merge_fields_for_put` tests

**File:** `tools/arrconf/tests/test_differ.py:87-131`

**Issue:** Three pre-existing tests (`test_merge_preserves_cluster_value_when_yaml_empty`,
`test_merge_handles_none_yaml_value`, `test_merge_mixed_some_empty_some_not`) construct cluster
`DownloadClient` instances with `fields=[{"name": "password", "value": "***REDACTED***"}]`
as plain dicts — without a `privacy` key. Because `FieldKV.privacy` defaults to `None`, the
omit-credential branch at line 140 never fires. The tests exercise the empty-value-preserve
path (D-31) as intended.

However, the docstrings and field names ("password", "username") imply these are credential
fields, which conflicts with the v0.1.5 contract (credential fields are omitted, not
preserved). A future maintainer adding `"privacy": "password"` to the fixture to make it more
realistic would silently invert the tested contract from "value preserved" to "field omitted."

**Status:** Still present, unchanged since previous review.

**Fix:** Add a clarifying comment on each of the three affected tests stating that the privacy
field is intentionally absent and the test is exercising the non-credential preserve path:

```python
def test_merge_preserves_cluster_value_when_yaml_empty() -> None:
    """D-31: cluster field has NO privacy metadata (privacy=None) and a non-empty value;
    YAML has '' — PUT body must carry the cluster value.

    NOTE: 'password' is used as the field name for convenience only. This test exercises
    the D-31 empty-value-preserve path for a field WITHOUT privacy metadata. If this
    field had privacy='password', it would instead be OMITTED by the v0.1.5
    omit-credential branch (see test_merge_fields_omits_privacy_password_when_value_is_api_mask).
    """
```

### WR-03: Stale docstring in `test_update_passes_forceSave_query_param`

**File:** `tools/arrconf/tests/test_reconcilers_sonarr.py:346-354`

**Issue:** The docstring for `test_update_passes_forceSave_query_param` states that
`?forceSave=true` is needed because `merge_fields_for_put` preserves the literal `"********"`
mask in the PUT body. After the v0.1.5 hotfix, `merge_fields_for_put` **omits** credential
fields entirely — the mask never reaches the PUT body. The stated rationale is now factually
incorrect.

**Status:** Still present. Not in scope of current file list (test_reconcilers_sonarr.py was
reviewed in the initial review pass but is excluded from this re-review's file list). The
finding stands as a maintenance debt.

**Fix:** Update the docstring to reflect the post-v0.1.5 reality:

```python
"""Regression — D-02.1-06 / ADR-8: every UPDATE PUT carries ``?forceSave=true``.

v0.1.3 rationale (now superseded): forceSave bypassed qBit auth re-validation
against the ``"********"`` mask that merge_fields_for_put previously preserved in
the PUT body.

v0.1.5 update: merge_fields_for_put now OMITS credential fields (privacy=password|
userName) entirely; the ``"********"`` mask no longer appears in the PUT body.
forceSave is retained as a defensive parameter — Sonarr's pre-save validation may
still reject PUT bodies for other field-type reasons during service restart windows.
"""
```

## Info

### IN-01: `getattr(current, "fields", [])` silently returns `[]` for future non-FieldKV models

**File:** `tools/arrconf/arrconf/differ.py:124-126`

**Issue:** The privacy-lookup at line 124-126 uses `getattr(current, "fields", [])` which
returns `[]` if the model does not have a typed `fields` attribute. In that case
`cur_privacy_by_name` is empty, no field is identified as a credential, and the old unsafe
preserve-cluster-mask behavior applies for all fields — reintroducing the bug for those model
types. This is safe today (Phase 1 only uses `DownloadClient` which has `fields: list[FieldKV]`),
but the D-33 reusability comment does not flag this constraint.

**Fix:** Add a comment at the `cur_privacy_by_name` line noting the precondition:

```python
# Precondition for D-33 reuse: models whose fields[] carry credential entries MUST
# declare `fields` as a typed `list[FieldKV]` pydantic attribute (not via extra='allow').
# If `fields` is absent or extra-only, getattr returns [] and no privacy omission occurs.
cur_privacy_by_name: dict[str, str | None] = {
    f.name: f.privacy for f in getattr(current, "fields", [])
}
```

the import appear as though it was meant to be hidden or is conditional.

**Fix:** Move the import to the module-level block at the top of the file alongside
`DownloadClient`:

```python
from arrconf.resources.sonarr.download_client import DownloadClient

from arrconf.resources.sonarr.download_client import DownloadClient, FieldKV
```

Then remove the local import at line 400.
