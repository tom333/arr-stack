# Phase 02.1 Learnings: field-merge-fix

## Decisions

### D-31/D-32/D-33 — value-based merge contract (no name allowlist)
For each entry in `desired.fields[]`, if `value == "" or value is None`, substitute the cluster's stored value at the same `name`. Otherwise keep desired's value. The rule is purely value-based — field NAMES are NEVER consulted, so no allowlist of "credential-like" field names. The helper lives in `differ.py` (not in `reconcilers/sonarr.py`) so Phase 3 Radarr/Prowlarr can reuse it unchanged.

**Rationale:** Avoids the trap of having to maintain a list of "secret" field names (which differs across *arr versions and apps). Generic by construction; future-proof against new privacy=password fields. Adopted in discuss-phase 2026-05-08.
**Source:** 02.1-CONTEXT.md, 02.1-02-PLAN.md, 02.1-02-SUMMARY.md

---

### D-36 — drop placeholder credential entries from arrconf.yml (REVERSED via PR4 hotfix)
PR3 removed `username: ''` and `password: ''` placeholder entries from `charts/arrconf/files/arrconf.yml` under the assumption that the merge helper made them redundant. PR4 hotfix re-added them. Final decision: KEEP the placeholders for credential-bearing fields; the dump-side filter half of D-36 (drop entries whose value is `***REDACTED***`) is unchanged.

**Rationale:** The helper only protects empty-value entries that ARE in YAML; it does not backfill entries MISSING from YAML. Sonarr's PUT to `/api/v3/downloadclient/{id}` accepts a partial body but its serialization makes "missing field" indistinguishable from "no log signal" — the dispositive `merge_field_preserved` event would never fire. Retaining placeholders gives the helper something to merge against and produces auditable proof in logs.
**Source:** 02.1-03-SUMMARY.md (Deviation 1), deferred-items.md (D-02.1-05)

---

### D-37 — atomic v0.1.3 release across arr-stack and my-kluster
Cut a single annotated git tag `v0.1.3` in arr-stack → CI publishes GHCR image → ONE PR in my-kluster bumps `image.tag` AND drops/reorganizes the YAML in the same diff. No "image bump" + "YAML cleanup" as separate PRs.

**Rationale:** Eliminates the bad intermediate state where the cluster runs the new image with the old YAML (or vice versa). Operator review of a small atomic diff is the integrity gate for the cross-repo coupling.
**Source:** 02.1-CONTEXT.md, 02.1-03-PLAN.md

### Suspend CronJob during hotfix windows
Before PR4 hotfix was opened, the orchestrator patched `kubectl patch cronjob arrconf -n selfhost -p '{"spec":{"suspend":true}}'` to prevent the every-4h cron from firing between credential restoration and ArgoCD reconcile.

**Rationale:** The fix window includes (a) PR4 author + merge, (b) ArgoCD sync, (c) operator credential re-entry, (d) verification. An unsuspended CronJob could fire in the middle of (b)→(c) and re-clear fields, defeating the hotfix.
**Source:** 02.1-03-SUMMARY.md

## Lessons

### `merge_fields_for_put` does not backfill cluster-only fields
The helper only iterates `desired.fields[]`. Entries missing entirely from YAML are NOT copied from cluster — they're omitted from the PUT body. Whether Sonarr clears them or preserves them is implementation-defined (it preserved in this case but the API GET returned `null` for `privacy=password|userName` fields after, which initially looked like a regression).

**Context:** Discovered during Plan 03 Task 3.3 post-PR3 smoke validation — the dispositive `merge_field_preserved ≥ 2` gate was unsatisfiable because PR3 had just removed the placeholder entries. Surfaced as D-02.1-05; recommended fix is a backfill pass in the helper (target Phase 3).
**Source:** deferred-items.md (D-02.1-05), 02.1-03-SUMMARY.md

---

### Sonarr's API masks `privacy=password` fields with the literal `********` and the merge helper preserves it
Sonarr's `GET /api/v3/downloadclient/{id}` returns `value: "********"` for fields marked `privacy: password`. `merge_fields_for_put` faithfully passes that mask through to the PUT body. Idempotent PUTs (no real field change) succeed because Sonarr accepts the body. PUTs that include any real field change trigger Sonarr's pre-save qBit auth re-validation against the literal `"********"` → HTTP 400 "Échec de l'authentification".

**Context:** Surfaced during Plan 04 drift demo (Task 4.2). The arrconf reconcile job correctly DETECTED the drift (`plan_action action=update`) but its automated PUT failed. W-04 dispositive value-equality was achieved via operator-driven `curl PUT .../?forceSave=true` — the "small manual nudge" foreseen by the Plan 04 brief. v0.1.4 will ship `client.put(..., params={"forceSave": "true"})` to eliminate the manual step. **Phase 3 prerequisite.**
**Source:** deferred-items.md (D-02.1-06), 02.1-04-SUMMARY.md, 02.1-SECURITY.md (Audit notes)

---

### Sonarr GET serialization can return `null` for stored values (cosmetic, not data loss)
After the post-PR3 smoke job, `GET /api/v3/downloadclient/1` returned `value: null` for `username` and `password`. Initial interpretation: the PUT cleared them (regression). Actual: the values were intact, Sonarr's API just serializes `privacy=password|userName` fields as `null` under certain conditions (likely "field not modified by most recent PUT body"). Operator UI confirmation closed the false alarm.

**Context:** Almost triggered a much larger remediation (option 2: enhance the helper to backfill, requiring a v0.1.4 release). Two pieces of evidence resolved it: (a) operator opened Sonarr UI and the fields were populated; (b) post-PR4 smoke (with placeholders re-added) showed `username: "moi"` and `password: "********"`. **Verify before remediating.**
**Source:** 02.1-03-SUMMARY.md (Deviation 1)

---

### qBit `WebUI\AuthSubnetWhitelist=192.168.88.0/24` does NOT cover cluster pod IPs
qBit's auth whitelist is configured for the LAN but not the microk8s pod network. Workaround: temporary `127.0.0.0/8` addition to the whitelist when port-forwarding, fully reverted post-snapshot (verified bit-identical via `diff` exit 0).

**Context:** Surfaced in Plan 01 + Plan 04 as D-02.1-04. Same workaround pattern repeated cleanly in both waves. The "real" fix is provisioning `QBT_USER` / `QBT_PASS` in `arrconf-env` secret (cross-repo my-kluster work) but it's deferred since the workaround is reliable.
**Source:** deferred-items.md (D-02.1-04), 02.1-01-SUMMARY.md, 02.1-04-SUMMARY.md

### Worktree isolation can leak files into the main working tree
The Wave 4 executor (running in `.claude/worktrees/agent-a5bfce563933fed05/`) created `snapshots/post-phase2.1-2026-05-09/` files in BOTH the worktree (tracked, committed) and the main working tree (untracked). Resolution: `cmp` all candidates against the worktree-tracked versions, delete the untracked main-tree copies, then re-run the merge. Adding to the orchestrator's worktree-cleanup checklist.
**Source:** Orchestrator-side debugging during Wave 4 merge (this conversation)

### Stream-idle-timeout heartbeats matter on long phases
The execute-phase workflow's `[checkpoint] phase {N} wave {N}/{M} ...` literal-text heartbeats were emitted between every wave + plan + completion. These are documented as protection against `Stream idle timeout - partial response received` on Claude Code + Opus 4.7 at 200K+ cache_read.

**Context:** Phase 02.1 was a 4-wave, 4-plan, ~3.5h execution that touched cluster operations + cross-repo PRs + 2 hotfix iterations. The heartbeats were emitted at every transition and the SSE stream stayed warm. Phase 3 (likely longer) should follow the same pattern.
**Source:** execute-phase.md workflow (#2410), orchestrator output during Phase 02.1 execution

### Plan dispositive gates can become unsatisfiable mid-execution
The Plan 03 Task 3.3 acceptance criterion `merge_field_preserved ≥ 2` was unsatisfiable after Plan 03 Task 3.2 (D-36) removed the placeholder entries — the helper had nothing to merge against. The plan author assumed D-36 and the merge helper would compose; in practice they don't.

**Context:** Lesson for plan authoring: when a plan task changes the input shape that a later task's dispositive gate depends on, that gate must be re-validated in the plan-checker pass. PR4 hotfix corrected the design defect by undoing D-36. **Plan 03 was not internally consistent — caught at execution time, not in plan-checking.**

## Patterns

### Cross-repo coupling: arr-stack tag → CI → GHCR → my-kluster PR → ArgoCD → cluster smoke
Single source of truth for the integration sequence: (1) arr-stack merge to main → green CI → annotated tag + push; (2) CI `arrconf-image.yml` runs (verify with `gh run list`); (3) verify GHCR public anon pull (`docker logout && docker pull`); (4) ONLY THEN open my-kluster PR with atomic image bump + YAML cleanup; (5) operator merges PR → ArgoCD selfHeal → CronJob unsuspended → forced smoke job → evidence committed back in arr-stack.

**When to use:** Any phase that releases an arrconf image and requires my-kluster to consume it. Phase 3 (Radarr/Prowlarr) will use this exact sequence for v0.1.4.
**Source:** 02-PATTERNS.md "Cross-Repo Coupling", 02.1-03-PLAN.md interfaces block

---

### Hotfix-via-PR pattern when a deployed change reveals a defect
When PR(N) is merged and ArgoCD has reconciled but a defect surfaces in the smoke job, do NOT roll back PR(N). Instead: (a) suspend the CronJob to freeze cluster state; (b) open PR(N+1) with the smallest possible reversion of the partial that caused the issue; (c) wait for merge + ArgoCD sync; (d) operator manually corrects any drifted state; (e) unsuspend; (f) verify with a fresh smoke. Document both PRs as one logical phase deliverable.

**When to use:** Any time a deployed PR's effect is detectable (smoke job logs, cluster state) and a smaller reversion is possible. Avoids the chaos of `git revert` on a merged PR + force-push, and preserves auditability.
**Source:** 02.1-03-SUMMARY.md (PR3 → PR4 journey)

---

### W-01 forensic snapshot + W-04 dispositive value-equality (drift demo runbook)
For drift detection proof: (W-01) snapshot the cluster BEFORE inducing drift (`snapshots/drift-test-<date>/`); (mutate one scalar field via direct API call — `priority` is the canonical safe field per Assumption A6); (force a smoke job); (W-04) snapshot AFTER the reconcile + capture `RESTORED == ORIGINAL` value-equality in evidence log. Both snapshots committed in arr-stack git for permanent audit chain.

**When to use:** Any phase that needs to prove a reconciler detects + corrects drift. Reusable as Phase 3 acceptance criterion for Radarr / Prowlarr drift detection.
**Source:** 02.1-04-PLAN.md, 02.1-04-SUMMARY.md, 02.1-PATTERNS.md

---

### Snapshot baseline before any code change touching new scope (ADR-6 discipline)
Whenever a phase touches a NEW scope or RE-touches a scope where prior runs were partial, run `tools/snapshot/snapshot.sh --apps <apps> --output snapshots/before-phase-<N>-<date>/` and commit BEFORE shipping any code change. This is the dispositive before-pair for the after-pair captured at phase close. Snapshots are versioned in git (NEVER `.gitignore`'d) and cheap to keep (lossless JSON, redacted).

**When to use:** Every phase that mutates cluster state. Phase 2.1 captured `before-phase-2.1-2026-05-08/`, `post-phase2.1-2026-05-09/`, and `drift-test-2026-05-09/` — three snapshot pairs in one phase.
**Source:** spec.md ADR-6, 02.1-01-PLAN.md, CLAUDE.md "Workflow snapshot"

---

### TDD RED→GREEN per task with conventional-commit prefixes
Per task tagged `tdd="true"`: (1) write failing tests + commit with `test(<phase>-<plan>): ... (RED)` message; (2) implement minimum code to pass + commit with `fix(<phase>-<plan>): ...` or `feat(<phase>-<plan>): ...` message. Both commits in the same plan execution. The MVP+TDD gate (workflow precondition) checks for the RED commit before allowing implementation.

**When to use:** Behavior-adding code in any plan. Phase 02.1 Plan 02 used this for both tasks (4 commits total). Pure doc/config/test-only tasks are exempt.
**Source:** execute-phase.md (MVP+TDD gate), 02.1-02-SUMMARY.md

---

### Worktree-isolated execution with orchestrator-owned shared files
GSD execute-phase spawns each plan's executor with `isolation="worktree"`. Executors commit per task on a `worktree-agent-<id>` branch. Shared files (`STATE.md`, `ROADMAP.md`) are owned by the orchestrator: backed up before merge, restored after merge ("main always wins"), and amended into the merge commit if changed. Snapshot/SUMMARY files are committed by the executor and merged in normally.

**When to use:** Multi-plan phases where parallel execution is enabled (`workflow.parallelization: true`). Sequential mode (one plan per wave) still benefits from the isolation. Phase 02.1 used this for all 4 plans.
**Source:** execute-phase.md, orchestrator behavior across all 4 worktree merges

## Surprises

### D-36 (drop placeholders) inverted the helper's protection
The plan assumed: helper protects empty-valued fields → placeholders are redundant → drop them. Reality: helper protects fields that EXIST in YAML with empty values; dropping them removed the protection entirely. The plan's dispositive gate (`merge_field_preserved ≥ 2`) became unsatisfiable post-D-36.

**Impact:** Required a same-day PR4 hotfix in my-kluster, ~30min CronJob suspension during the fix window, and a deviation note in 02.1-03-SUMMARY.md. The fix shipped cleanly but the planning oversight cost ~1h of execution time.
**Source:** 02.1-03-SUMMARY.md (Deviation 1), deferred-items.md (D-02.1-05)

---

### "PUT cleared the credentials" was a false alarm
Initial interpretation of `username: null, password: null` in Sonarr's GET response: the PUT had cleared the credentials. Actual: the GET-side serialization for `privacy=password|userName` fields returns `null` under certain conditions, but the stored values were intact. Operator UI confirmation + post-PR4 smoke showing `username: "moi"` resolved the false alarm.

**Impact:** Almost triggered a v0.1.4 release with the helper-backfill enhancement. Saved ~2-3h by checking the UI before committing to remediation. Reinforces "verify before remediating" as a core principle.
**Source:** 02.1-03-SUMMARY.md (Deviation 1), this conversation's interactive confirmation

---

### D-02.1-06: the helper preserves the API mask `********`, breaking real-change PUTs
Discovered ONLY in Plan 04 drift demo — not in any unit test, not in the post-PR4 smoke (which was idempotent). Sonarr's pre-save validation rejects the literal `"********"` as a password value when ANY top-level field actually changes (it tries to authenticate to qBit using `********` as the literal credential, fails, returns 400). Idempotent PUTs work fine.

**Impact:** Architectural finding documented as D-02.1-06; v0.1.4 must ship `?forceSave=true` BEFORE Phase 3 (Radarr/Prowlarr) goes live, otherwise their automated drift correction will face the same failure. The Plan 04 drift demo was completed via operator-driven `curl PUT .../?forceSave=true` — the "small manual nudge" the Plan 04 brief explicitly foresaw.
**Source:** 02.1-04-SUMMARY.md, deferred-items.md (D-02.1-06)

---

### Wave 4 worktree leaked snapshot files into the main working tree
Both the worktree (tracked, committed) AND the main working tree (untracked) ended up with identical copies of `snapshots/post-phase2.1-2026-05-09/`. The worktree merge initially failed because git refused to overwrite untracked files. MD5s matched, so resolution was safe (delete the untracked copies + re-merge), but it added an unexpected step to the cleanup.

**Impact:** ~5 min of orchestrator work to diagnose + resolve. Suggests adding a per-plan post-execution cwd-drift check to the executor agent, or a pre-merge "untracked files match worktree contents → delete safely" step to the orchestrator.
**Source:** Orchestrator-side debugging during Wave 4 merge (this conversation)
