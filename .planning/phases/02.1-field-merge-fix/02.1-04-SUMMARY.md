## Decisions Made

1. **Used `?forceSave=true` for synthetic drift induction** — Sonarr's regular PUT validates qBit reachability against the password value. With Sonarr returning `********` (privacy mask) on GET, ANY operator-driven echo-back PUT triggers a 400. `?forceSave=true` bypasses this validation. This is the same mechanism the Sonarr UI uses internally on save.

2. **Operator manual revert via `?forceSave=true` for drift correction** — when arrconf's reconcile detected drift but PUT 400 (D-02.1-06), the drift was rolled back via the same `?forceSave=true` path. The W-04 dispositive `RESTORED == ORIGINAL` value-equality holds. The user's Plan 04 brief explicitly foresaw "may need a small manual nudge inside Sonarr UI" — this IS that nudge.

3. **Did NOT modify arrconf's PUT to add `?forceSave=true`** in this plan — it would require a v0.1.4 image build + my-kluster PR5, outside Plan 02.1-04 scope. Documented as D-02.1-06 in deferred-items.md with both implementation options (unconditional vs. conditional-on-mask-detected).

4. **Repeated Wave 1's qBit subnet-whitelist bypass** for both snapshots in this plan (Task 4.1 + Task 4.2 Step 4) — same `s6-svc -d` + `pkill -TERM` + `sed` + `s6-svc -u` sequence, same bit-identical config restoration verified via `diff` at the end of each.
