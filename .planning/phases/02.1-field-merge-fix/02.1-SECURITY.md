### Audit notes (Phase 3 awareness)

- **D-02.1-06 architectural follow-up (NOT a STRIDE threat — correctness gap):** `merge_fields_for_put` faithfully preserves Sonarr's GET-time `"********"` mask for `privacy=password` fields per the D-31 contract. Idempotent PUTs work fine (Sonarr accepts the body when no real change). Drift-correction PUTs that include a real field change trigger Sonarr's pre-save qBit auth re-validation against the literal `"********"` → HTTP 400 "Échec de l'authentification". This caused the drift-demo job to log all dispositive events correctly (`plan_action action=update`, `merge_field_preserved` ×2) but fail the actual PUT. Operator-driven `curl -X PUT .../?forceSave=true` (the "small manual nudge" foreseen in Plan 04 brief) completed the revert and satisfied W-04 value-equality. **Phase 3 prerequisite:** ship v0.1.4 with `client.put(path, id=..., json=body, params={"forceSave": "true"})` (or conditional on detected mask) BEFORE Radarr/Prowlarr reconcilers go live, otherwise their automated drift correction will face the same failure mode. See `deferred-items.md` D-02.1-06 for full reproduction recipe + ADR-style rationale + two implementation options.
- **D-02.1-05 (correctness, not security):** `merge_fields_for_put` does not backfill cluster-only fields missing from `desired.fields[]`. PR4 hotfix retains `username: ''` / `password: ''` placeholders in `arrconf.yml` to give the helper something to merge against. Phase 3 retro should evaluate adding a `merge_field_backfilled` event + corresponding test scenario.
- **D-02.1-01, D-02.1-02 (security, deferred to snapshot.sh enhancement):** `snapshot.sh` does not redact Sonarr `config_host` apiKey/password/username, nor qBit tracker passkey. Workaround applied per snapshot via manual `jq`. Recommend Phase 3 (or earlier) ships a `snapshot.sh` redaction pass; current snapshots in this phase are clean (49 + 50 + 50 = 149 REDACTED placeholders applied across the three snapshot dirs).
