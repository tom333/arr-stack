<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator browser → arrconf-ui backend | LAN-trusted, single-operator; only crosses to the Phase 25 `/api/configarr/*` file-read/write endpoints |
| arrconf-ui → *arr APIs | MUST NOT exist (ADR-5) — the frontend reads/writes the config file only, never an *arr API URL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-01 | Tampering | FieldInput readOnly bypass | accept | readOnly is a UI affordance, not a security control. The authoritative guard is the Phase 25 backend (D-04 api_key opaque, D-09 tag-literal guard) which preserves `!env`/`!secret` tags on PUT regardless of frontend payload. A LAN-trusted single operator editing a disabled field has no privilege boundary to cross. Low risk. |
| T-26-02 | Information disclosure | api_key rendered in form | accept | api_key is stored as a `!secret`/`!env` literal in configarr.yml; the Phase 25 backend never resolves it, so the value shown is the tag literal, not a resolved secret (Phase 25 SC#4). Read-only display of a tag literal on a LAN-trusted tool is acceptable. |
| T-26-03 | Elevation of privilege | *arr API URL construction in frontend | mitigate | ADR-5 boundary: FieldInput introduces no fetch/URL building. Enforced by the acceptance-criteria grep gate (`grep -c "fetch(\|/api/v3\|http://"` returns 0) in Task 2. |
| T-26-04 | Tampering | XSS via config values rendered in widgets | accept | Svelte escapes interpolated values by default; widget values bind to `value`/`title` attributes (no `{@html}`). configarr.yml is operator-authored on a local tool — no untrusted input source. Low risk. |

No HIGH-severity threats. Block gate (high) not triggered.
