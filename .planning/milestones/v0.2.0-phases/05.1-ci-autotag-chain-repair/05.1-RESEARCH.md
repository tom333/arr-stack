## Summary

Phase 5.1 requires exactly two GitHub Actions workflow file edits totalling ~15-20 lines of YAML. The root cause of the broken chain is confirmed and well-understood: `mathieudutour/github-tag-action` pushes tags via `GITHUB_TOKEN`, and GitHub's anti-loop policy suppresses downstream `on: push: tags:` triggers from the same token. This is a known problem (see metadata-action Discussion #503 [CITED: github.com/docker/metadata-action/discussions/503]).

The fix is `repository_dispatch`: the `tag` job in `chart-lint.yml` fires a POST to the GitHub Dispatches API after creating the tag; `arrconf-image.yml` listens on `repository_dispatch: types: [arrconf-image-build]` and rebuilds with the payload tag. The critical technical detail is how `docker/metadata-action@v5` extracts the semver — under `repository_dispatch`, `github.ref` is `refs/heads/main`, not a tag ref. The action's `type=semver` processor will produce no semver tag unless given a `value=` override explicitly. The planner MUST use the `value=` attribute pattern.

**Primary recommendation:** In `arrconf-image.yml`, pass the tag to `docker/metadata-action@v5` via `type=semver,pattern={{version}},value=${{ github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref }}`. Check out the correct commit via `actions/checkout@v4` with `ref:` set to the same expression. This is the single most important implementation detail.

### A2 — `docker/metadata-action@v5` semver extraction under `repository_dispatch`

**This is the most critical technical finding.**

**Claim:** Under `repository_dispatch`, `github.ref` = `refs/heads/main`. The `type=semver` tag type reads `github.ref` by default and will NOT produce a semver tag when the ref is a branch, only when it is a tag ref of the form `refs/tags/vX.Y.Z`.

**VERIFIED by codebase inspection:** `arrconf-image.yml` line 34 is `type=semver,pattern={{version}}` with no `value=` attribute. Under the current `on: push: tags:` trigger, `github.ref = refs/tags/v0.2.1` → action strips `refs/tags/` prefix → `v0.2.1` → strips leading `v` → `0.2.1`. This is correct. Under `repository_dispatch`, `github.ref = refs/heads/main` → no semver match → the `type=semver` line produces no output tag. [CITED: github.com/docker/metadata-action/discussions/503] (this exact scenario, confirmed in discussion).

**The fix (VERIFIED):** `docker/metadata-action@v5` supports an explicit `value=` attribute on any tag type. [CITED: github.com/docker/metadata-action README] The correct syntax:

```yaml
type=semver,pattern={{version}},value=${{ github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref }}
```

When `client_payload.tag = "v0.3.1"`, this evaluates to `value=v0.3.1` → action strips leading `v` → produces Docker tag `:0.3.1`. When `github.ref = refs/tags/v0.3.1` (legacy push-tags trigger), `client_payload.tag` is absent → falls through to `github.ref` → same semver extraction as before.

**Checkout ref:** `actions/checkout@v4` must also be pointed at the tag commit:

```yaml
- uses: actions/checkout@v4
  with:
    ref: ${{ github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref }}
```

Without this, the build context would be the HEAD of `main` at trigger time, not the tagged commit. In practice for this project these are the same commit (tags are created on the same push), but aligning is correct practice.

**`latest` tag guard:** The existing line `type=raw,value=latest,enable=${{ startsWith(github.ref, 'refs/tags/v') }}` uses `github.ref`. Under `repository_dispatch`, `github.ref = refs/heads/main` → `startsWith` is false → `:latest` is NOT emitted. This is the correct behavior for non-tag-push triggers. [ASSUMED: no change needed to the `latest` enable condition since repository_dispatch should NOT push `:latest` — only manual or legacy tag-push should.]

### A3 — `secrets.GITHUB_TOKEN` scope for `repository_dispatch`

**VERIFIED:** The GitHub Dispatches API (`POST /repos/{owner}/{repo}/dispatches`) requires the calling token to have `contents: write` permission (or the legacy `repo` scope for PATs). [CITED: github.com/peter-evans/repository-dispatch — the peter-evans action, which calls the same endpoint, documents this requirement explicitly.]

The `tag` job in `chart-lint.yml` already declares:
```yaml
permissions:
  contents: write   # lines 138-139 confirmed via codebase inspection
```

This permission is what `mathieudutour/github-tag-action` uses to push the tag. The same token can also call the Dispatches API. **No permission changes needed.**

**Dispatch call syntax (recommended — gh CLI):**

```yaml
- name: Dispatch arrconf-image build
  env:
    GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
  run: |
    gh api -X POST repos/${{ github.repository }}/dispatches \
      -f event_type=arrconf-image-build \
      -f "client_payload[tag]=${{ steps.tag.outputs.new_tag }}"
```

`gh` CLI is pre-installed on all GitHub-hosted runners [VERIFIED: github.com/actions/runner-images]. The `GH_TOKEN` env var is the standard auth mechanism for `gh api`. This approach avoids adding a new action dependency.

**Alternative — `peter-evans/repository-dispatch@v3`:**

```yaml
- uses: peter-evans/repository-dispatch@v3
  with:
    token: ${{ secrets.GITHUB_TOKEN }}
    event-type: arrconf-image-build
    client-payload: '{"tag": "${{ steps.tag.outputs.new_tag }}"}'
```

Both approaches are equivalent. Recommend the `gh api` approach (no new action dependency, simpler audit surface).

### A4 — Side effects on existing triggers

**VERIFIED by codebase inspection of `arrconf-image.yml`:**

- **GHCR login:** `if: github.event_name != 'pull_request'` — evaluates to `true` for `repository_dispatch` (event_name = `repository_dispatch`) and `workflow_dispatch` (event_name = `workflow_dispatch`). GHCR login WILL fire for both new triggers. Correct behavior.
- **Docker push:** `push: ${{ github.event_name != 'pull_request' }}` — same evaluation. Push WILL fire. Correct.

- **branch tag:** `type=ref,event=branch,prefix=branch-` — under `repository_dispatch`, there is no branch ref event, so this produces no output. Existing `:branch-main` tag from push-on-main runs is unaffected.

**No side effects on push-on-main or PR triggers.** The PR trigger condition (`on: pull_request: paths: ['tools/arrconf/**']`) is unchanged.

- `push: branches: main` → value empty → reads `refs/heads/main` → no semver tag (expected) ✓

## Edge Cases

### Edge Case 1: `new_tag` empty — tag action doesn't bump

`mathieudutour/github-tag-action@v6.2` with `default_bump: patch` will ALWAYS create a new tag on push-to-main, even with no conventional commits. [CITED: github.com/mathieudutour/github-tag-action — `default_bump` parameter docs: "default bump to use when no bump keyword is found in commit messages"]. The `new_tag` output will be empty ONLY if the action is configured with `default_bump: false` AND no commit contains a bump keyword. In the current config (`default_bump: patch`), `new_tag` will always be non-empty on a successful run.

However, the `if: steps.tag.outputs.new_tag != ''` guard is still recommended as defensive coding — costs nothing, prevents a spurious dispatch with an empty tag if behavior ever changes.

### Edge Case 2: Dispatch API rate-limit or transient 5xx

The `gh api` call will exit non-zero on any HTTP error, causing the `tag` job to fail. Per the discussion in CONTEXT.md D-05.1-SCOPE-01, the tag job should NOT fail just because the dispatch step fails (the tag itself is already created and that's the irreversible part).

**Recommendation:** Wrap the dispatch step with `continue-on-error: true`. If the dispatch fails, the tag still exists and the operator can use `workflow_dispatch` (rescue path, D-05.1-RESCUE-01) to manually trigger the image build.

```yaml
      - name: Dispatch arrconf-image build
        if: steps.tag.outputs.new_tag != ''
        continue-on-error: true            # Don't fail the tag job if dispatch API blips
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
        run: |
          gh api -X POST repos/${{ github.repository }}/dispatches \
            -f event_type=arrconf-image-build \
            -f "client_payload[tag]=${{ steps.tag.outputs.new_tag }}"
```

### Edge Case 3: Two PRs merge in quick succession → two dispatches

Two parallel `arrconf-image.yml` builds on GHCR. Docker push is content-addressable — both builds produce the same image digest for the same code, and the second push simply overwrites the manifest pointer with an identical digest. No race condition risk. [ASSUMED: both build from the same checkout ref; if tags differ (v0.3.1 vs v0.3.2), they produce separate image tags with no collision.]

### Edge Case 4: `push: tags: ['v*']` still present (D-05.1-LEGACY-01)

When someone pushes a tag manually from a workstation with a PAT, `arrconf-image.yml` triggers via the legacy `push: tags:` path. Under this path, `github.ref = refs/tags/v0.3.1`, `client_payload.tag` is absent, `inputs.tag_ref` is absent. The `value=` expression evaluates to empty string → metadata-action reads `github.ref` → `:0.3.1` produced correctly. No double-build issue because the dispatch step in chart-lint.yml only fires when the tag is created by the auto-tag job (`github.event_name == 'push' && github.ref == 'refs/heads/main'`) — a manual tag push directly to a tag ref would bypass chart-lint entirely.

## CI Guard Recommendation (D-05.1-GUARD-01 / Q-05.1-1)

**Recommendation: ADD the guard.** Cost: 3 lines in `chart-lint.yml`. Risk averted: a future PR that accidentally removes `repository_dispatch:` from `arrconf-image.yml` would silently re-break the chain with no visible failure. This exact failure mode is what created Phase 5.1.

```yaml
      - name: "arrconf-image chain guard (repository_dispatch must remain in arrconf-image.yml)"
        run: |
          grep -q "repository_dispatch:" .github/workflows/arrconf-image.yml || \
            { echo "ERROR: repository_dispatch: missing from arrconf-image.yml — chain would break"; exit 1; }
          echo "OK: repository_dispatch trigger present"
```

**Place:** in the `lint` job of `chart-lint.yml`, alongside the existing guards (`:latest` guard, `cronJobConfig:` guard). This is the correct placement because (a) it runs on both PRs and push-to-main, (b) it catches the regression before the `tag` job fires.

**Trigger path:** `chart-lint.yml` already has `paths: ['.github/workflows/chart-lint.yml']` in its trigger — **but NOT `.github/workflows/arrconf-image.yml`**. Add `'.github/workflows/arrconf-image.yml'` to the `push: paths:` and `pull_request: paths:` lists so the guard runs when either file changes. Without this, a PR that only modifies `arrconf-image.yml` would not trigger `chart-lint.yml` at all.

## Common Pitfalls

### Pitfall 1: `type=semver` produces no output under non-tag ref
**What goes wrong:** `docker/metadata-action@v5` `type=semver` silently emits zero tags when `github.ref` is not a tag ref. The build succeeds, the push succeeds (with sha + branch tags), but `:0.3.1` never appears on GHCR.
**Why it happens:** This is the root cause of the original break. The action's semver processor only fires on `refs/tags/vX.Y.Z` patterns unless `value=` is provided.
**How to avoid:** Always use `value=<explicit>` for `type=semver` when the trigger is not `push: tags:`. Verify SC#4 (GHCR manifest probe) after the first post-merge run.
**Warning signs:** Image build CI run succeeds but GHCR shows only `:sha-<hash>` and `:branch-main` — no `:0.3.1`.

### Pitfall 2: Dispatch step failing silently blocks image chain
**What goes wrong:** If `continue-on-error: true` is absent, a rate-limit or transient API error on the dispatch call fails the `tag` job — meaning the auto-tag already created the git tag but no image build is triggered AND the tag job is marked failed.
**Why it happens:** `gh api` exits non-zero on any non-2xx response.
**How to avoid:** Use `continue-on-error: true` on the dispatch step. The tag is the durable artifact; the dispatch is best-effort. Operators use `workflow_dispatch` as the recovery path.

### Pitfall 3: `chart-lint.yml` trigger doesn't include `arrconf-image.yml` path
**What goes wrong:** The CI guard step (SC#6) never runs on PRs that only modify `arrconf-image.yml`, because `chart-lint.yml`'s `push/pull_request: paths:` list doesn't include that file.
**How to avoid:** Add `'.github/workflows/arrconf-image.yml'` to both `push: paths:` and `pull_request: paths:` in `chart-lint.yml`.

### Pitfall 4: `new_tag` output used without null-guard
**What goes wrong:** If `new_tag` is empty (edge case), the dispatch call sends `client_payload.tag=` (empty string), causing metadata-action to produce no semver tag.
**How to avoid:** `if: steps.tag.outputs.new_tag != ''` on the dispatch step.
