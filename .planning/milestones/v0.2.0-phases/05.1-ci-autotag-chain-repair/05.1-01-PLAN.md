<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner -> GitHub Dispatches API | `secrets.GITHUB_TOKEN` (scope: contents:write in chart-lint.yml `tag` job) posts to `/repos/{owner}/{repo}/dispatches` - fully in-repo, no external secret exchange |
| chart-lint.yml `tag` job -> arrconf-image.yml workflow trigger | repository_dispatch event payload `client_payload.tag` is consumed as a checkout ref and metadata-action `value=`. Trusted because emitted by same-repo workflow under same `GITHUB_TOKEN` |
| arrconf-image.yml build -> GHCR | Existing trust boundary (unchanged) - `packages: write` permission, image push under `${{ github.actor }}` |
| Future PR review -> workflow file integrity | A future PR could remove `repository_dispatch:` from arrconf-image.yml and silently re-break the chain - mitigated by D-05.1-GUARD-01 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05.1-DISPATCH-FAIL | Denial of Service | chart-lint.yml `tag` job dispatch step | mitigate | `continue-on-error: true` on the dispatch step so transient API errors (rate-limit / 5xx) do not fail the `tag` job. The tag is the durable artifact; `workflow_dispatch` rescue path (D-05.1-RESCUE-01) is always available for manual recovery. |
| T-05.1-METADATA-SILENT | Tampering (silent build success without semver image - exact root cause of D-05-CI-AUTOTAG-CHAIN) | arrconf-image.yml `meta` step | mitigate | Explicit `value=${{ github.event.client_payload.tag \|\| github.event.inputs.tag_ref \|\| github.ref }}` on `type=semver` line so the action does not silently emit zero semver tags under repository_dispatch (where `github.ref = refs/heads/main`). Post-merge SC#4 GHCR HEAD probe (Plan 05.1-02) catches any regression. |
| T-05.1-SILENT-REGRESS | Tampering (a future PR removes repository_dispatch trigger and silently re-breaks the chain) | arrconf-image.yml `on:` block | mitigate | CI regression guard step in chart-lint.yml `lint` job: grep checks for both `repository_dispatch:` and `workflow_dispatch:` in arrconf-image.yml; exit 1 if missing. Also extend chart-lint paths to include arrconf-image.yml so the guard runs on changes to either file (Pitfall 3 mitigation). |
| T-05.1-EMPTY-TAG | Tampering (auto-tag no-op emits empty `new_tag` -> dispatch sends `client_payload.tag=""` -> metadata-action falls back to `github.ref=refs/heads/main` -> no semver image) | chart-lint.yml `tag` job dispatch step | mitigate | `if: steps.tag.outputs.new_tag != ''` guard on dispatch step. Tag step is given `id: tag` so `steps.tag.outputs.new_tag` resolves. |
| T-05.1-CHECKOUT-REF | Tampering (workspace at HEAD-of-main when dispatch lags behind a subsequent commit -> image built from wrong code) | arrconf-image.yml `actions/checkout@v4` step | mitigate | `with: ref: ${{ github.event.client_payload.tag \|\| github.event.inputs.tag_ref \|\| github.ref }}` - checkout the exact tagged commit, not whatever HEAD-of-main looks like at trigger time. Legacy `push: tags:` path falls through to `github.ref = refs/tags/vX.Y.Z` unchanged. |

Block-on threshold: HIGH severity. None of these are HIGH (no auth bypass, no data exfil, no destructive ops). All mitigations are in-plan static checks executed at PR time + post-merge GHCR probe. Plans pass security gate as drafted.

Two modifications inside the `build` job in `.github/workflows/arrconf-image.yml`:

**(a) Update `actions/checkout@v4`** (currently `- uses: actions/checkout@v4` at line 18) to switch the workspace to the dispatched tag:

```yaml
      - uses: actions/checkout@v4
        with:
          ref: ${{ github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref }}
```

Rationale (T-05.1-CHECKOUT-REF): under `repository_dispatch` the default `github.ref = refs/heads/main`, but we want to build the EXACT tagged commit. The fallback chain handles all trigger paths:
- `repository_dispatch` with `client_payload.tag = "v0.3.1"` -> checkout that tag
- `workflow_dispatch` with `tag_ref = "v0.3.1"` -> checkout that tag
- `workflow_dispatch` with no `tag_ref` -> `github.ref` (= `refs/heads/main`)
- `push: tags: ['v*']` -> `github.ref` (= `refs/tags/vX.Y.Z`)
- `push: branches: main` -> `github.ref` (= `refs/heads/main`)

**(b) Update `docker/metadata-action@v5`** (currently lines 27-35). Replace the `tags:` block with:

```yaml
      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/tom333/arr-stack-arrconf
          tags: |
            type=sha,prefix=sha-,format=short
            type=ref,event=branch,prefix=branch-
            type=semver,pattern={{version}},value=${{ github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref }}
            type=raw,value=latest,enable=${{ startsWith(github.event.client_payload.tag || github.event.inputs.tag_ref || github.ref, 'refs/tags/v') || startsWith(github.event.client_payload.tag || github.event.inputs.tag_ref || '', 'v') }}
```

Why each piece (THIS IS THE CRITICAL FIX - Pitfall 1 / T-05.1-METADATA-SILENT):
- `type=sha` + `type=ref,event=branch` UNCHANGED.
- `type=semver,pattern={{version}},value=${{ ... }}` is the entire technical content of the fix. Without `value=`, the action reads `github.ref`, which is `refs/heads/main` under `repository_dispatch` -> NO semver tag emitted -> exact root cause of D-05-CI-AUTOTAG-CHAIN. With `value=v0.3.1` from the payload, the action strips leading `v` -> produces `:0.3.1`.
- `type=raw,value=latest,enable=...` extended: the OR with `startsWith(payload || input || '', 'v')` ensures `:latest` is published when a `v*` tag is built via `repository_dispatch` OR `workflow_dispatch`, not just legacy push-tags. The first `startsWith(github.ref, 'refs/tags/v')` handles the legacy push-tags path.
