CONTEXT

Plan 05-08 Task 8.3 dispositive: operator manually adds an anime series via
Sonarr UI with tag `anime` → series fetched via Prowlarr/indexer → qBit
downloads to /data/anime/<series>/ → Sonarr imports to /media/anime/<series>/.

PRE-REQUISITE FIX APPLIED (not in Phase 5 plan — operator deviation)
---------------------------------------------------------------------
Initial attempt in Sonarr UI failed with "Impossible de communiquer avec
qBittorrent. Failed to connect to qBittorrent, check your settings."

Root cause: the 3 new Phase-5 download clients (Sonarr id=2/3/4 — qBittorrent
- TV/Anime/Family) were POSTed by arrconf with empty username + password
fields (chart YAML uses `value: ''` per CLAUDE.md "Ne pas committer de
secrets"). Phase 2.1's `merge_fields_for_put` helper preserves cluster values
on UPDATE but not on CREATE — so credentials were stored as literal empty
strings.

Workaround: 6 download clients (Sonarr × 3 + Radarr × 3) patched via Sonarr
/ Radarr API:
  PUT /api/v3/downloadclient/{2,3,4}?forceSave=true
  body: GET response with fields[name=username].value=$QBT_USER + password=$QBT_PASS

All 6 returned HTTP 202 (forceSave=true accepted per ADR-8 trusted-controller
pattern). Connectivity test then green:
  POST /api/v3/downloadclient/test for each id:
    qBittorrent - TV (Sonarr id=2)     → HTTP 200 {}
    qBittorrent - Anime (Sonarr id=3)  → HTTP 200 {}
    qBittorrent - Family (Sonarr id=4) → HTTP 200 {}
    qBittorrent - Movies (Radarr id=2) → HTTP 200 {}
    qBittorrent - Anime (Radarr id=3)  → HTTP 200 {}
    qBittorrent - Family (Radarr id=4) → HTTP 200 {}
