## Phase Boundary

Single-objective interrupt phase between Phase 2.1 and Phase 3. Add `?forceSave=true` to arrconf's UPDATE-branch PUT so Sonarr's pre-save validation does not re-authenticate to qBittorrent using the literal `"********"` API mask preserved by `merge_fields_for_put`. Closes D-02.1-06 architectural finding from Phase 2.1 — required prerequisite before Phase 3 (Radarr/Prowlarr automated drift correction would face the identical 400 failure mode).

**In scope:**
- Code change in `tools/arrconf/arrconf/client_base.py` (HTTP client layer) so every UPDATE PUT carries `?forceSave=true` unconditionally for *arr v3 clients (Sonarr now; Radarr/Prowlarr inherit in Phase 3)
- New unit test `test_update_passes_forceSave_query_param` (final name TBD — D-02.1-06 proposed `test_update_uses_forceSave_when_masked_credential_in_body` but Option A unconditional makes the broader assertion correct)
- New structured log event `put_force_save_used` matching the `merge_field_preserved` / `managed_tag_created` audit pattern from Phase 2.1
- New ADR-8 in `spec.md` §11: "arrconf is a trusted controller — bypasses *arr UI-grade pre-save validation via `?forceSave=true` on UPDATE PUTs"
- Image release `v0.1.4` on GHCR via the cross-repo coupling pattern (single annotated tag → CI → GHCR → ONE atomic my-kluster PR bumping `image.tag: 0.1.3 → 0.1.4`)
- Pre-deploy snapshot baseline (`snapshots/before-phase-2.2-2026-05-09/`) per ADR-6
- Post-deploy verification: forced smoke job confirming PUT 200 in a normal reconcile cycle
- Re-execution of Plan 02-05/02.1-04 drift demo runbook with W-01 forensic snapshot before + W-04 dispositive value-equality on `priority` after, this time WITHOUT the operator-driven manual `curl ?forceSave=true` nudge — closes the correction half of REQ-drift-detection cleanly

**Out of scope (deferred):**
- Phase 3 reconciler additions (Radarr, Prowlarr, indexers, notifications, root_folders, tags, host_config) — Phase 3 picks up the inherited forceSave default from `client_base.py`
- Backfill behaviour in `merge_fields_for_put` for fields missing entirely from YAML (D-02.1-05) — separate finding; Phase 2.2 only fixes the mask-passthrough half
- my-kluster YAML changes — `username: ''` / `password: ''` placeholders STAY (per Phase 2.1 PR4 hotfix decision); v0.1.4's only diff in my-kluster is the `image.tag` bump

<decisions>
## Implementation Decisions

### Trigger Condition
- **D-02.2-01:** `forceSave=true` is applied **unconditionally** on every UPDATE PUT for *arr v3 clients (Sonarr/Radarr/Prowlarr). Trigger is the action (UPDATE), not the body content.
  - Rationale: matches Sonarr's React UI behavior on Save click. Eliminates the "did we forget the kwarg?" failure mode for Phase 3+ reconcilers. Keeps the code simple — no body scan, no mask-alphabet sync, no coupling between HTTP layer and merge layer's logging.
  - Trade-off acknowledged: if Sonarr ever introduces a meaningful pre-save check we'd actually want, arrconf bypasses it. Counter: arrconf already merges cluster state into the PUT body (D-31/D-32), so pre-save validation re-runs are a tax with no signal value in arrconf's controller context.
  - ADD and DELETE actions are unaffected. Only the UPDATE branch (`Action.UPDATE` in `differ.py`) carries forceSave.
  - Rejected: Option B (conditional on mask in body) — adds a body scan and mask-alphabet maintenance burden for negligible safety gain. Rejected: Option C (conditional on `merge_field_preserved` events fired) — couples HTTP layer to merge-layer logging.

### Implementation Site
- **D-02.2-02:** forceSave lives at the HTTP **client layer** in `tools/arrconf/arrconf/client_base.py`, not at the reconciler `_execute` layer.
  - Rationale: Phase 3 will add `RadarrClient` and `ProwlarrClient` (and Phase 6/7 will add `SeerrClient`/`JellyfinClient`). Putting the kwarg in `_execute` means every new *arr-v3 reconciler must remember `params={"forceSave": "true"}` — failure mode of forgetting it = the same 400 we just fixed. Client-layer default is inherited by construction.
  - Phase 3 reconcilers inherit forceSave automatically. No "don't-forget-this-kwarg" rule for new reconcilers.
  - Test asserts the URL params at the HTTP layer (respx `put_route.calls.last.request.url.params["forceSave"] == "true"`).
  - Rejected: Reconciler `_execute` UPDATE branch — explicit but creates a per-reconciler-discipline rule. Rejected: helper in `differ.py` alongside `merge_fields_for_put` — invents an abstraction for one kwarg.

### Claude's Discretion (planner picks)
- **Architectural shape of the inheritance tree:**
  - Option (a) Add `forceSave=true` as a default `params` in `ArrApiClient.put()` and have non-*arr-v3 subclasses (`qBittorrentClient`, `JellyfinClient` in later phases) override `put()` to drop it.
  - Option (b) Introduce a `_ArrV3Client(ArrApiClient)` intermediate / mixin that overrides `put()` to inject forceSave; `SonarrClient`/`RadarrClient`/`ProwlarrClient` inherit from it.
  - Option (c) Override `put()` directly in `SonarrClient` for Phase 02.2 and let Phase 3 promote to a shared parent when `RadarrClient`/`ProwlarrClient` arrive.
  - Planner's call based on cohesion + minimum surface area. (b) is most explicit about scope; (a) is least code change today; (c) defers a decision but risks the same forgotten-kwarg failure mode at Phase 3 boundary.
- **Final test name** — D-02.1-06 proposed `test_update_uses_forceSave_when_masked_credential_in_body`, but with Option A unconditional the broader assertion is `every UPDATE PUT carries forceSave=true regardless of body content`. Suggest `test_update_pass_forceSave_query_param` (or similar). Planner's call.
- **`put_force_save_used` event payload shape** — at minimum `{path, id}`. Whether to also include `diff_fields` (which would require threading data from the reconciler into the client) is a planner judgment call; cluster logs already capture the parent `plan_action action=update diff_fields=[...]` event.
- **Test coverage scope** — at minimum the new unit test asserting forceSave on UPDATE. Whether to also assert the `put_force_save_used` event is emitted, and whether to add a test asserting forceSave is NOT present on ADD or DELETE (defensive), is a planner call. The Phase 2.1 regression test (`test_update_preserves_credentials`) already covers UPDATE behavior; the new assertion can extend it or stand alone.
- **Whether v0.1.4 also rolls in the D-02.1-05 helper backfill** — D-02.1-05 says `merge_fields_for_put` does not backfill cluster-only fields (entries missing entirely from YAML). Adding backfill to v0.1.4 is plausible (same "drift correction safety" theme) but expands scope. Default: NO — keep v0.1.4 single-purpose (forceSave only). If planner thinks composition is cleaner, escalate before expanding scope.

### Verification Depth
- **D-02.2-03:** Verification re-runs the **full drift demo dispositive**, not just a happy-path smoke.
  - Sequence: (1) pre-deploy snapshot baseline (`snapshots/before-phase-2.2-2026-05-09/sonarr/`); (2) v0.1.4 release + my-kluster PR + ArgoCD sync; (3) forced smoke job confirming PUT 200 on a normal reconcile cycle (post-deploy snapshot diff = 0 except expected noise); (4) drift demo runbook — W-01 forensic snapshot before, mutate `priority` via operator `curl PUT .../?forceSave=true` (priority is the canonical safe drift field per Assumption A6), force smoke job, capture log + W-04 dispositive `RESTORED_PRIORITY == ORIGINAL_PRIORITY` value-equality, this time WITHOUT a manual `curl ?forceSave=true` nudge from the operator.
  - Closes REQ-drift-detection's correction half cleanly (Phase 2.1 closed it via manual nudge — Phase 2.2 closes it via automated reconcile).
  - Adds ~30-45 min of cluster work but locks the architectural finding shut before Phase 3.
  - Snapshots committed in arr-stack git per ADR-6 (NEVER `.gitignore`'d).
  - Rejected: smoke-only — leaves Phase 3 starting with REQ-drift-detection still nominally PARTIAL until its first reconcile lands. Rejected: unit-only — leaves v0.1.4 image unverified in cluster context until Phase 3, packaging/CI regressions discovered under Phase 3 pressure.

### Audit & Documentation
- **D-02.2-04:** Add **ADR-8** to `spec.md` §11 documenting the architectural stance: "arrconf is a trusted controller — bypasses *arr UI-grade pre-save validation via `?forceSave=true` on UPDATE PUTs."
  - Captures the constraint for Phase 3+ contributors. Phase 3 (Radarr/Prowlarr), Phase 5 (qBittorrent — explicitly out of scope for forceSave), Phase 6 (Seerr — TBD), Phase 7 (Jellyfin — TBD) all need to know this stance is in effect for *arr v3 PUTs.
  - The ADR explains WHY the bypass is safe in arrconf's context: (a) arrconf's own `diff` already determined a real change is needed; (b) the merged PUT body preserves cluster's stored credentials by design (D-31/D-32 contract); (c) arrconf is the trusted controller — Sonarr's UI-grade validation is meant for human-driven misconfigs.
- **D-02.2-05:** Emit a **`put_force_save_used`** structured log event in `client_base.py.put()` matching the `merge_field_preserved` / `managed_tag_created` audit pattern from Phase 2.1.
  - Makes the bypass auditable in cluster JSON logs alongside the existing `plan_action action=update` event.
  - Payload at minimum `{path, id}`; additional fields at planner's discretion.

### Inherited from Phase 2.1 (no re-asking)

## Specific Ideas

- **Test name calibration**: D-02.1-06 proposed `test_update_uses_forceSave_when_masked_credential_in_body`. With Option A unconditional locked, the broader assertion `every UPDATE PUT carries forceSave=true regardless of body content` is the correct assertion — suggest `test_update_passes_forceSave_query_param` or `test_put_carries_forceSave_for_arr_v3` at the client_base test level. Planner picks final name.
- **Drift field choice**: re-use `priority` (canonical safe field per Assumption A6 from Phase 2.1 Plan 02-05). Same field used in Phase 2.1 W-04 dispositive — provides direct comparability between the two drift demo runs (manual-nudge vs fully-automated).
- **W-04 dispositive shape**: `RESTORED_PRIORITY == ORIGINAL_PRIORITY` value-equality logged via structlog. Phase 02.2's run is FULLY AUTOMATED (no operator `curl PUT .../?forceSave=true` mid-run); Phase 2.1's run had the manual nudge. The differential between the two runs is the dispositive proof that v0.1.4 closes the correction half of REQ-drift-detection.
- **Snapshot pair naming**: `snapshots/before-phase-2.2-2026-05-09/` (pre-deploy baseline) and `snapshots/post-phase2.2-<DATE>/` (post-drift-demo). Mirrors Phase 2.1's `before-phase-2.1-2026-05-08/` + `post-phase2.1-2026-05-09/` convention.
- **PUT params override mechanism**: `client.put()` callers should be able to override or merge additional `params` if a future caller needs them — i.e. don't hard-code `params={"forceSave": "true"}` in a way that loses caller-supplied params. Either merge dicts inside the method or use a default that callers can replace.
