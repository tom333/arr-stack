## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Plan Step 6 anonymous manifest probe used too-narrow Accept header set, returning 404 against an OCI image index**

- **Found during:** Task 4.2 Step 6 (sanity-check via `curl` that GHCR confirms tag is public)
- **Issue:** The plan's Step 6 used Accept header `application/vnd.docker.distribution.manifest.v2+json` only. The GHCR-published image is wrapped in a multi-arch OCI image index (even though the build only emits `linux/amd64`); the index has media type `application/vnd.oci.image.index.v1+json`, not the legacy Docker manifest list. With only the Docker `manifest.v2` Accept, GHCR's content-negotiation returned 404 (after the expected anon-bearer 401 → token → retry round-trip).
- **Fix:** Added the OCI Accept types to the curl request: `application/vnd.docker.distribution.manifest.list.v2+json`, `application/vnd.oci.image.manifest.v1+json`, and `application/vnd.oci.image.index.v1+json` — broad Accept set that lets GHCR negotiate to whatever media type the published artifact uses. Result: 200 with the OCI index returned; the `etag` and `docker-content-digest` headers both match the dispositive `docker pull` digest captured in Step 2 (`sha256:1e7e60c4d0af432366af026ab48e33f911ba6ee1cdc3f25167afbd846cd6054a`).
- **Files modified:** None (the fix was inline in shell command construction, not in a tracked file)
- **Commit:** None (no source-tree changes — the dispositive verification still passed with the broader headers)
- **Why this is Rule 1, not Rule 4:** The plan's verification mechanism was technically incorrect for the modern GHCR/OCI artifact shape; broadening the Accept header is a probe-tooling fix, not an architectural reconsideration. The dispositive evidence (`docker pull`) was already in hand and was always the load-bearing assertion — the curl probe is corroborating, not primary.
- **Pattern established:** Phase 3 release verification Probe pattern updated — any future GHCR anonymous-manifest probe in arr-stack release plans MUST include the OCI index Accept type, even for single-platform builds, because GHCR wraps single-platform images in a 1-entry index.
