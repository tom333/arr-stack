```python
def put(self, path: str, id: int, json: Any, **kwargs: Any) -> Any:
    """HTTP PUT /{path}/{id} — returns parsed JSON of updated resource."""
    return self._request("PUT", f"{path}/{id}", json=json, **kwargs).json()
```

**Existing `_request()` retry+kwargs forwarding (lines 67-82) — UNCHANGED:**
```python
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=10),
    retry=retry_if_exception_type((httpx.NetworkError, httpx.TimeoutException, ServerError)),
    reraise=True,
)
def _request(self, method: str, path: str, **kwargs: Any) -> httpx.Response:
    response = self._client.request(method, path, **kwargs)
    if response.status_code == 401:
        raise AuthError(f"{self.name}: 401 — check API key")
    if response.status_code == 404:
        raise NotFoundError(f"{self.name}: 404 — {method} {path}")
    if 500 <= response.status_code < 600:
        raise ServerError(f"{self.name}: {response.status_code} — {response.text[:200]}")
    response.raise_for_status()
    return response
```

The retry decorator already wraps `_request`; whatever `params=` reaches `_client.request()` inherits the retry naturally. No retry-layer change needed.

**Pattern shape — three architectural options (planner picks):**

#### Option (a) — default `params` on `ArrApiClient.put()`, scoped via class flag

Add a class attribute `force_save_on_put: bool = False` to `ArrApiClient`; flip it to `True` in `SonarrClient` (and Phase 3's `RadarrClient`/`ProwlarrClient`). Mutate `kwargs["params"]` inside `put()` only when the flag is set. Caller-supplied params are merged, never clobbered (per CONTEXT.md "Specifics §PUT params override mechanism").

```python
class ArrApiClient:
    api_path: str = "/api/v3"
    name: str = "arr"
    force_save_on_put: bool = False  # *arr v3+: True (Sonarr/Radarr/Prowlarr); qBit/Jellyfin: False
    # ... existing __init__, _request unchanged ...

    def put(self, path: str, id: int, json: Any, **kwargs: Any) -> Any:
        """HTTP PUT /{path}/{id} — returns parsed JSON of updated resource."""
        if self.force_save_on_put:
            params = dict(kwargs.pop("params", None) or {})  # caller-supplied wins on conflicts
            params.setdefault("forceSave", "true")
            kwargs["params"] = params
            log.info("put_force_save_used", path=path, id=id)
        return self._request("PUT", f"{path}/{id}", json=json, **kwargs).json()


class SonarrClient(ArrApiClient):
    api_path = "/api/v3"
    name = "sonarr"
    force_save_on_put = True  # D-02.2-01 / ADR-8: arrconf is a trusted controller
```

**Trade-offs (a):**
- ✅ Smallest diff (~10 LOC). One class attribute + 4 new lines in `put()`.
- ✅ Phase 3's `RadarrClient`/`ProwlarrClient` opt-in via `force_save_on_put = True` line.
- ❌ Failure mode at Phase 5 (`qBittorrentClient`) and Phase 7 (`JellyfinClient`): if a contributor forgets to keep `force_save_on_put = False`, those clients would inherit a *arr-specific URL param. Documented inverse default (`False` on the base) mitigates.

#### Option (b) — `_ArrV3Client(ArrApiClient)` intermediate class

Insert a mixin/intermediate that overrides `put()`. `SonarrClient` (and Phase 3's Radarr/Prowlarr) inherit from `_ArrV3Client`; `qBittorrentClient`/`JellyfinClient` keep inheriting from `ArrApiClient` directly.

```python
class _ArrV3Client(ArrApiClient):
    """Common base for *arr v3 REST clients (Sonarr/Radarr/Prowlarr).

    Always sends ``forceSave=true`` on UPDATE PUTs (D-02.2-01 / ADR-8).
    arrconf is the trusted controller — Sonarr's UI-grade pre-save validation
    (which re-authenticates against masked passwords) is a tax with no signal
    value in our context.
    """

    def put(self, path: str, id: int, json: Any, **kwargs: Any) -> Any:
        """HTTP PUT /{path}/{id} with ``forceSave=true`` always set."""
        params = dict(kwargs.pop("params", None) or {})
        params.setdefault("forceSave", "true")
        kwargs["params"] = params
        log.info("put_force_save_used", path=path, id=id)
        return self._request("PUT", f"{path}/{id}", json=json, **kwargs).json()


class SonarrClient(_ArrV3Client):  # ← parent changed
    api_path = "/api/v3"
    name = "sonarr"
```

**Trade-offs (b):**
- ✅ Most explicit about scope: a class named `_ArrV3Client` is self-documenting. New contributors at Phase 5/7 see the type hierarchy and know not to inherit from it.
- ✅ Phase 3 just inherits from `_ArrV3Client` — zero discipline burden.
- ❌ Slightly more code (a new class). One additional layer in the type hierarchy.

#### Option (c) — override `put()` directly in `SonarrClient` only

Keep `ArrApiClient` unchanged. Override `put()` on `SonarrClient`. Promote to a shared parent in Phase 3 when Radarr/Prowlarr arrive.

**Trade-offs (c):**
- ❌ Defers the same architectural decision to Phase 3 — creates the exact "forgot-the-kwarg" failure mode CONTEXT.md D-02.2-02 warns against (Phase 3's `RadarrClient` would silently miss `forceSave` until someone notices).
- ❌ Duplicate code at Phase 3 if not promoted in time.
- **Recommendation: avoid.** Pick (a) or (b).

#### Recommendation summary

The `02.1-PATTERNS.md` precedent recommends Option (a)-equivalent for `merge_fields_for_put` (extended `differ.py` rather than creating `merge.py`). By the same cohesion argument, Option (a) here is smallest. But Option (b) is the only one that surfaces the architectural scope ("this is *arr v3 territory") at the type level — which is what Phase 5/7 contributors will read first.

**Planner picks based on:** clarity-vs-size trade-off; the more-explicit option (b) is preferable if the planner expects Phase 5/7 contributors to read `client_base.py` cold.

**Pitfalls (apply to all three options):**
- **Caller-supplied `params=` must win on conflicts.** Use `setdefault` after copying caller's dict, NOT `params = {"forceSave": "true"}` which would clobber. CONTEXT.md "Specifics §PUT params override mechanism".
- **Only UPDATE.** ADD (`post`) and DELETE (`delete`) are unaffected. The fix lives in `put()` — ADD/DELETE methods stay byte-identical.
- **`forceSave` value is the literal string `"true"`** — not the Python `True`. Sonarr's URL-param parser expects lowercase string. Verified by `02.1-04-SUMMARY.md` operator-driven `curl PUT .../?forceSave=true` — that exact string worked.
- **Logging happens once per PUT call.** Place `log.info("put_force_save_used", ...)` inside `put()`, NOT inside `_request` (where retries would emit duplicates).
- **mypy strict on signatures** (CLAUDE.md): the new class attribute `force_save_on_put: bool = False` is type-annotated; the override method signatures must match `ArrApiClient.put()` exactly (or use `# type: ignore[override]` if drift is intentional, but it should not drift).

```python
def configure_logging(level: str = "INFO") -> None:
    """Configure structlog with TTY-aware rendering (D-07).

    JSON output for non-TTY (cluster CronJob log pipeline), pretty colored
    console output for TTY (local dev DX).
    """
    log_level = getattr(logging, level.upper(), logging.INFO)
    is_tty = sys.stderr.isatty()
    processors: list[Any] = [
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.StackInfoRenderer(),
        structlog.dev.set_exc_info,
    ]
    if is_tty:
        processors.append(structlog.dev.ConsoleRenderer(colors=True))
    else:
        processors.append(structlog.processors.JSONRenderer())
    structlog.configure(
        processors=processors,
        wrapper_class=structlog.make_filtering_bound_logger(log_level),
        cache_logger_on_first_use=True,
    )
```

**Pattern shape for ADR-8 (per CONTEXT.md D-02.2-04):**
```markdown
### ADR-8 — arrconf is a trusted controller — bypasses *arr UI-grade pre-save validation

**Contexte** : Sonarr (et par extension Radarr/Prowlarr en *arr v3+) effectue une pré-validation côté serveur sur tout PUT modifiant une ressource avec credentials (download_client.password, indexer.apiKey, etc.). Cette pré-validation est conçue pour rattraper les misconfigs UI : elle re-authentifie contre le service externe (qBittorrent, indexer) en utilisant la valeur littérale du champ. Côté arrconf, le helper `merge_fields_for_put` (Phase 2.1, D-31) préserve le mask `********` pour les champs `privacy=password` parce que arrconf YAML ne porte jamais de vrai secret (CLAUDE.md "Variables d'environnement"). Résultat : le PUT atomique d'arrconf échoue en 400 dès qu'un champ top-level change vraiment, parce que Sonarr essaie d'auth qBit avec `********` comme mot de passe.

**Décision** : tout client *arr v3 d'arrconf (Sonarr, Radarr, Prowlarr) envoie systématiquement `?forceSave=true` sur les PUT UPDATE. Cela demande à Sonarr de skipper la pré-validation et d'écrire la ressource telle quelle. arrconf est un **trusted controller** — le scénario que la pré-validation protège (humain qui se trompe via UI) ne s'applique pas.

**Raisons** :
- arrconf a déjà décidé qu'un changement réel est nécessaire via son propre `diff` (idempotence golden rule, CLAUDE.md)
- Le body PUT est construit pour préserver les credentials du cluster par design (D-31/D-32 — value-based merge contract)
- Le pre-save validation Sonarr est une pré-validation UX pour utilisateurs humains ; arrconf n'est pas un humain
- Sans `forceSave=true`, la moitié correction de REQ-drift-detection est inatteignable sans intervention opérateur manuelle (constaté Phase 2.1 W-04 dispositive — D-02.1-06)
- Centraliser au layer HTTP client (D-02.2-02) plutôt qu'au layer reconciler évite le mode d'échec "j'ai oublié le kwarg" pour Phase 3+

**Conséquences** :
- Un nouveau log event `put_force_save_used` (event=info, payload `{path, id}`) est émis à chaque UPDATE PUT — auditable côté cluster
- Phase 3 Radarr/Prowlarr héritent par construction du comportement (option a class flag, ou option b mixin `_ArrV3Client`) — pas de discipline contributeur
- qBittorrent (Phase 5) et Jellyfin (Phase 7) ont des sémantiques PUT différentes ; ils restent OUT OF SCOPE de `forceSave` (leur classe parent garde le défaut `False`)
- Si Sonarr introduit un jour un check pre-save vraiment utile, arrconf le skippera. Trade-off acceptable : les checks utiles seraient une régression upstream, pas une feature ; on les détectera par la perte d'idempotence (qui resterait visible dans les logs `plan_action`)

**Alternatives rejetées** :
- **Trigger conditionnel sur mask dans body** (Option B de D-02.1-06) : ajoute un body scan + maintenance d'un alphabet de mask Sonarr (`********`, `***REDACTED***`, autres ?) — coût de maintenance inutile pour un gain de sécurité négligeable (la pré-validation étant inutile dans le contexte arrconf de toute façon)
- **Trigger conditionnel sur événements `merge_field_preserved` émis** : couple le layer HTTP au layer merge — inversion de responsabilité
- **Override au layer reconciler `_execute`** : crée une discipline par-reconciler ("n'oublie pas `params={"forceSave": "true"}`") — exactement le mode d'échec qu'on veut éviter pour Phase 3
```

**Pitfalls:**
- The `Conséquences` block must mention **the qBittorrent/Jellyfin scope-out** explicitly — CONTEXT.md "Deferred" notes this is a writing-style call, but it's load-bearing context for Phase 5/7 contributors and worth surfacing here.
- Keep ADR-8's prose in **French** — ADR-1 through ADR-7 are all in French (project convention).
