<decisions>
## Implementation Decisions

### Empty-Value Semantic
- **D-31:** In arrconf YAML, `value: ''` (or `value: null`) for any download_client field MEANS "preserve whatever Sonarr/cluster has stored". It does NOT mean "set to empty string". This is the contract change that fixes the Phase 2 PUT 400.
  - Rationale: CLAUDE.md is explicit that secrets only travel via env at runtime; arrconf YAML can never legitimately carry a real secret value. So the only meaningful interpretation of an empty string in YAML is "I don't manage this field; preserve cluster value".
  - Implication: cluster's stored credential survives every round-trip (`dump → apply` is a no-op).

### Detection Rule
- **D-32:** "Preserve from cluster" is triggered by **empty value only** — `value == ''` or `value is None`. Field name is NOT consulted.
  - Rationale: simplest possible rule. No allowlist to maintain across reconcilers. Any field can be opted out by setting it empty in YAML.
  - Trade-off: if user wants a field to truly be empty in cluster, they'd need a different signal (out of scope for 2.1; if it ever comes up, add an explicit sentinel like `__EMPTY__`).

### Fix Scope (Generalize)
- **D-33:** Implement the merge logic in a **shared helper** (`differ.py` or a new `arrconf/merge.py`) so Phase 3's Radarr + Prowlarr reconcilers inherit it. Same `fields[]` pattern (array of `{name, value, type}` objects) exists in all *arr v3+ APIs.
  - Rationale: the bug is generic; fixing it Sonarr-only sets up Phase 3 to re-rediscover the same issue.
  - Boundary: keep helper narrow — only handles the `fields[] with name/value pairs` pattern, not arbitrary array merging.

### Phase 2 Closure Inside 2.1
- **D-34:** Re-run Plan 02-05 Tasks 5.1c (post-PR2 snapshot + Sonarr API tag verification) + 5.2 (drift demo runbook) **inside Phase 2.1**, after the v0.1.3 image deploys to cluster.
  - Closes ROADMAP success criteria #4 (download_client gets `arrconf-managed` tag id attached) + #5 (drift demo with W-04 dispositive value-equality on `priority`).
  - Closes Phase 1 HUMAN-UAT #3 (live round-trip) and 02-05 Deviation §1 ("Task 5.1c PARTIAL").
  - Phase 3 starts with a clean slate, not a backlog of Phase 2 carry-forwards.

### Test Strategy
- **D-35:** Add **respx unit tests** for the merge logic (multiple scenarios: empty preserved, non-empty overridden, mixed, edge cases) PLUS a **dump-and-merge round-trip test** that asserts: load YAML → simulate cluster current state with redacted credentials → build PUT body → assert credential fields equal cluster's stored values, not YAML's empty.
  - Explicitly NOT adding: Sonarr-container-in-CI integration test (rejected — too much CI complexity for marginal confidence). Cluster smoke job (Plan 02-05 Task 5.1c re-execution) IS the integration signal.
  - Explicitly NOT adding: hypothesis property-based tests (overkill for a 2-day patch).

### YAML Cleanup
- **D-36:** PR3 in my-kluster removes the `username: ''` and `password: ''` field entries from `charts/arrconf/files/arrconf.yml` entirely (cleaner — under D-31 they'd be no-ops anyway). The `arrconf dump` command must also stop emitting field entries whose Sonarr-side value is REDACTED (to keep round-trip clean).
  - Rationale: empty entries in committed YAML risk future confusion ("is this a leak? a placeholder?"). Better to omit and let the new "preserve" semantic apply implicitly.
  - Implication: arrconf's dump output will be smaller and won't show sensitive field names at all for download_client `fields[]`.

## Existing Code Insights

### Reusable Assets
- `differ.py:diff_models()` (lines 50-54) — already excludes `_READ_ONLY_FIELDS`; the merge helper can hook into a similar pattern for "preserve from current"
- `differ.py:reconcile()` (lines 57-106) — produces `PlannedAction[T]` with `current` AND `desired`; the data needed for merging is already on the planned action object
- `sonarr.py:_execute` UPDATE branch (lines 93-101) — clear, isolated; merge step inserts here as `body = merge_for_put(p.current, p.desired)` instead of `body = p.desired.model_dump(...)`
- `tests/conftest.py` — respx fixtures for Sonarr API; new tests reuse the patterns
- `tests/test_round_trip.py` — round-trip pattern already exists; extend rather than create

### Established Patterns
- **6-case Action enum** (differ.py L28-36) — fix only affects UPDATE path; ADD/DELETE/NO_OP/PRUNE_* are unchanged
- **`exclude_none=True` in model_dump** — currently used; merge logic must consider this (None values dropped, empty strings retained — that's the asymmetry causing the bug)
- **`arrconf-managed` tag id** (sonarr.py D-02) — the desired body's `tags` list is built from cluster's tag id (Pitfall 1 — IDs not names); merge logic must NOT clobber `tags` (whose desired value comes from arrconf reconciler logic, not from YAML)
- **`_READ_ONLY_FIELDS` set** (differ.py L19-25) — already excludes `id`, `implementationName`, `infoLink`, `message`, `presets` — sensitive credential field names could be added here, but D-32 says rule is value-based not name-based, so this set stays as-is

### Integration Points
- The fix changes ONE method's behavior (`_execute` UPDATE branch), so blast radius is contained
- Phase 1's existing tests should still pass without modification (they use respx mocks that didn't exercise empty-value scenarios)
- The `tags` field path needs careful handling — desired's `tags` list legitimately overrides cluster's (when adding `arrconf-managed`); the fix must whitelist `tags` as a "use desired" field while everything else inside `fields[]` becomes "preserve from current if value empty"
- Phase 3's Radarr/Prowlarr reconcilers will import the same helper from `differ.py` (or wherever D-33 places it)

</code_context>

<specifics>
## Specific Ideas

- Approach to merging follows the natural `PlannedAction[T]` shape: the helper takes `current: T | None` (always non-None for UPDATE) + `desired: T` and returns the merged dict ready for PUT
- The `fields[]` array merge is by `name` key (matching the `name/value/type` Sonarr API shape): for each field-by-name in desired, take YAML value if non-empty, else cluster's value
- Round-trip test mental model: `cluster_state = {fields: [{name:'username', value:'admin'}, {name:'password', value:'***REDACTED***'}, ...]}` + `yaml = {fields: [{name:'username', value:''}, {name:'password', value:''}, ...]}` → merged PUT body should have `value: 'admin'` and `value: '***REDACTED***'` for those fields (i.e., cluster's values survive)

</specifics>
