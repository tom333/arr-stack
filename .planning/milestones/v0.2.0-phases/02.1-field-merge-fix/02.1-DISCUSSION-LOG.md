## Empty-Value Semantic

| Option | Description | Selected |
|--------|-------------|----------|
| Preserve cluster value | Empty in YAML = "this field is NOT arrconf-managed; keep whatever Sonarr already has". Aligns with CLAUDE.md 'secrets only via env'. Fixes the bug semantically. | ✓ |
| Set to empty string (current/broken) | Current behavior. PUT writes empty string, Sonarr 400s on credential fields. Not viable. | |
| Treat as null and skip from PUT body | Pydantic exclude_none would drop the entry. But Sonarr's API may require all fields present in fields[] array. | |
| Other | Free-text. | |

**User's choice:** Preserve cluster value
**Notes:** Sets up D-31. Locks the contract: `value: ''` always means "preserve, don't overwrite". Round-trip stays clean (`dump → apply` is a no-op).

## Detection Rule

| Option | Description | Selected |
|--------|-------------|----------|
| Empty-value-only | If YAML value is '' or null → preserve cluster's value. Field names not consulted. Simple, predictable, no maintained list. | ✓ |
| Name-based allowlist | Hardcode sensitive names: username, password, apiKey, passkey, token, secret. Independent of value. Drawbacks: list to maintain across reconcilers. | |
| Both (empty OR sensitive name) | Belt-and-braces. Preserve if value empty OR name matches sensitive list. Most robust, slightly more code. | |
| Heuristic (regex on field name) | Detect via regex on field name (any field matching /password\|secret\|key\|token/i). Auto-extends as APIs add new sensitive fields. | |

**User's choice:** Empty-value-only
**Notes:** Sets up D-32. Simplest possible rule. Trade-off acknowledged: if user ever wants a field to truly be empty in cluster, an explicit sentinel would be needed (deferred — see Deferred Ideas in CONTEXT.md).
