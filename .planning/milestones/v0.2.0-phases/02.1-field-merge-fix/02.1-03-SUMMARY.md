## Deviations

### Deviation 1 — D-36 design defect surfaced; PR4 hotfix required

**Discovered during** Task 3.3 of this plan, while validating the post-PR3 smoke job.

**Root cause:** `merge_fields_for_put` (Wave 2) only iterates fields that are PRESENT in `desired.fields[]` (i.e., in arrconf.yml). For each, if value is empty, it substitutes the cluster's stored value. But fields that are **missing entirely** from `desired.fields[]` are not protected — they're simply omitted from the PUT body.

PR3 (D-36) removed the `username: ''` and `password: ''` placeholder entries from `arrconf.yml`. After the merge, the PUT body to `/api/v3/downloadclient/1` no longer contained those entries. Sonarr's PUT semantics for `downloadclient` accepted the partial body without clearing the missing fields (the API reports `null` for `privacy=password|userName` fields under certain conditions, which initially looked like a regression but turned out to be cosmetic — the stored values were never actually cleared, as confirmed by the operator and the post-PR4 verification showing `value: "moi"` and `value: "********"`).

However, the plan's dispositive gate was unsatisfiable: with no empty entries in YAML, `merge_field_preserved` would never fire, and the only proof that the helper was active end-to-end would be an inference (PUT 200 instead of 400) — not a positive log signal.

**Resolution:** PR4 hotfix re-adds the placeholders so the helper sees empty values, fires `merge_field_preserved`, and substitutes cluster's stored credentials at PUT time. This is the design intent of the merge helper as specified in D-31/D-32/D-33. The post-PR4 smoke captured the dispositive `merge_field_preserved` events for both username and password.

**Why we did NOT enhance the helper to backfill missing entries:** That would be a larger code change (new event `merge_field_backfilled`, new tests, potentially a v0.1.4 release). For Phase 2.1's scope, sticking with the helper's existing contract — "protect empty entries that ARE in YAML" — is the smallest correct change. Phase 3 (Radarr / Prowlarr) is a natural place to revisit whether `merge_fields_for_put` should also backfill for true field-name-agnostic safety, since those reconcilers will face the same shape.

**Consequence on D-36:** D-36 (drop placeholder entries) is effectively REVERTED for credential-bearing fields. The dump-side filter half of D-36 (drop entries whose value is `***REDACTED***`) is unchanged and still applies. Future arrconf YAML for credential-bearing fields should keep the placeholder entries. Documenting this in `.planning/phases/02.1-field-merge-fix/deferred-items.md` (D-02.1-05) for Phase 3 retro.

### Deviation 2 — Smoke job exit-code check uses succeeded count

The plan's Task 3.3 Step 1 had `[ "$EXIT_CODE" = "1" ]` checking `.status.succeeded == 1` (which is the count of succeeded pods). That's correct, but I read it twice before realizing the variable name "$EXIT_CODE" is misleading. No code change needed — the gate worked as intended.
