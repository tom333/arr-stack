<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator workstation -> Sonarr / Radarr / Prowlarr / qBit / Seerr / Jellyfin (port-forward) | Read-only API GETs; trust = workstation kubeconfig holder |
| arr-stack repo (commit) | Lossless JSON dumps committed; no plaintext credentials in payloads (the upstream APIs already redact secrets when GET-ing config endpoints) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure (T-secret-leak) | snapshots/ JSON files committed in arr-stack | mitigate | The snapshot.sh script captures only GET responses; *arr/qBit/Seerr/Jellyfin APIs do not return plaintext API keys in their config endpoints — fields like `apiKey` come back redacted ("apiKey": "***" or empty). Audited in Phase 0 commit. CLAUDE.md confirms "Ne pas redacter le contenu JSON — snapshots sont lossless raw API output" — meaning the upstream redaction is what we rely on. |
| T-02-02 | Denial of Service | port-forwards leak (process accumulation) | accept | Port-forwards are local processes; if the operator forgets to kill them, the next snapshot still works (the pf is idempotent). Cleanup is `pkill -f 'kubectl.*port-forward'`. |
| T-02-03 | Tampering | snapshot script bug or partial capture | mitigate | Acceptance criteria explicitly count JSON files per app subdir (>= 5 for full *arr; >= 1 for qBit/Seerr where the API surface is smaller). If any app dir is empty, executor must investigate the port-forward / API key for that service before continuing. |
</threat_model>
