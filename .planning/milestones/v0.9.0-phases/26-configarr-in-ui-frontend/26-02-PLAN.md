    2. Load helper — extract a `loadForConfig(cfg: ActiveConfig)` that dispatches by config
       and resets schema/configState/savedConfig (mirror the onMount body):
       ```typescript
       async function loadForConfig(cfg: ActiveConfig) {
         schema = null; configState = null;
         try {
           const [s, c] = cfg === 'arrconf'
             ? await Promise.all([api.getSchema(), api.getConfig()])
             : await Promise.all([api.getConfigarrSchema(), api.getConfigarrConfig()]);
           schema = s;
           configState = c as typeof configState;
           savedConfig = JSON.parse(JSON.stringify(c));
           loadError = null;
         } catch (e) {
           loadError = e instanceof Error ? e.message : String(e);
         }
       }
       ```
       Replace the onMount body with `await loadForConfig('arrconf');`.

    3. Tab switch + confirm gate (D-04) — gate on the existing `diffCount` dirty state:
       ```typescript
       function requestTabChange(next: ActiveConfig) {
         if (next === activeConfig) return;
         if (diffCount > 0) { pendingSwitch = next; confirmSwitchOpen = true; return; }
         void doSwitch(next);
       }
       async function doSwitch(next: ActiveConfig) {
         activeConfig = next; confirmSwitchOpen = false; pendingSwitch = null;
         await loadForConfig(next);
       }
       function cancelSwitch() { confirmSwitchOpen = false; pendingSwitch = null; }
       ```

    4. Save pipeline — dispatch postDiff/putConfig by activeConfig:
       - In `openDiffPanel`: `const resp = activeConfig === 'arrconf' ? await api.postDiff(configState) : await api.postConfigarrDiff(configState as Record<string, unknown>);`
       - In `confirmSave`: `const resp = activeConfig === 'arrconf' ? await api.putConfig(configState) : await api.putConfigarrConfig(configState as Record<string, unknown>);`
       Keep the existing ApiError → validationErrors handling unchanged.

    5. HeaderBar mount (lines 97-102) — drive filePath + tabs from state:
       ```svelte
       <HeaderBar
         filePath={CONFIG_FILE_PATHS[activeConfig]}
         {diffCount} {saveStatus}
         onSaveClick={openDiffPanel}
         {activeConfig}
         onTabChange={requestTabChange}
       />
       ```

    6. Section rendering (D-05) — render CategoriesEditor and the SectionDoc("categories")
       block ONLY when `activeConfig === 'arrconf'` (configarr has no categories concept).
       For the section loop, choose the section list by config. CRITICAL: `AppSection`
       only handles the `additionalProperties` dict-of-instances pattern (the
       sonarr/radarr → ArrInstance shape). The configarr schema's top-level `properties`
       ALSO contains scalar/array keys — `trashGuideUrl` (string), `recyclarrConfigUrl`
       (string), `customFormatDefinitions` (array) — that have NO `additionalProperties`
       and would render as empty "0 champs" collapsibles (broken UX, contradicts D-05's
       clean schema-driven list). So the configarr section list MUST be filtered to keys
       whose schema node carries `additionalProperties` — that leaves exactly sonarr +
       radarr, where quality_profiles / custom_formats / per-profile scores live (the
       CFGUI-04 scope). KISS: a single `.filter` on `additionalProperties`, no new
       component:
       ```svelte
       {@const configarrSections = Object.keys(schema.properties).filter(
         (k) => (schema.properties[k] as { additionalProperties?: unknown }).additionalProperties != null
       )}
       {@const sectionNames = activeConfig === 'arrconf' ? APP_SECTIONS : configarrSections}
       {#each sectionNames as sectionName}
         {@const sectionSchema = schema.properties[sectionName]}
         {#if sectionSchema}
           <SectionDoc section={sectionName} />
           <AppSection {sectionName} {sectionSchema} root={schema}
             value={configState[sectionName] as Record<string, unknown>}
             onChange={(next) => updateAppSection(sectionName, next)}
             errors={validationErrors} />
         {/if}
       {/each}
       ```
       `trashGuideUrl` and `recyclarrConfigUrl` are intentionally NOT editable in this
       phase — they are file-level scalar config (URLs), consistent with the readOnly
       philosophy of keeping TRaSH/configarr-owned plumbing out of the form.
       `customFormatDefinitions` (top-level array) is likewise out of CFGUI-04 scope; the
       editable custom_formats live per-instance under sonarr + radarr.

       readOnly fields (api_key / media_naming / quality_definition) render disabled +
       lock badge automatically via the FieldInput readOnly support from 26-01 (the marker
       is on the schema node) — no per-field handling here.

    7. Unsaved-switch confirm dialog — render a simple confirm UI when `confirmSwitchOpen`,
       using `UNSAVED_SWITCH_MESSAGE`, with "Changer" (calls `doSwitch(pendingSwitch!)`)
       and "Annuler" (calls `cancelSwitch`) buttons. Match the existing DiffPanel/dialog

    8. Loading/error copy — the hardcoded "arrconf.yml" strings in the loadError message

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator browser → arrconf-ui backend | LAN-trusted single operator; crosses only to Phase 25 `/api/config*` and `/api/configarr/*` file endpoints |
| arrconf-ui → *arr APIs | MUST NOT exist (ADR-5); App.svelte performs no URL construction, only calls api.ts functions |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-05 | Tampering | configarr.yml tag corruption on save (`!env`/`!secret` drop) | mitigate | The frontend sends the parsed config to PUT; the Phase 25 backend (D-09 tag-literal guard, ruyaml round-trip) preserves `!env`/`!secret` tags verbatim. Verified in the checkpoint via `git diff` (SC#3 step 4b). Frontend introduces no tag handling of its own. |
| T-26-06 | Elevation of privilege | *arr API URL construction in App.svelte | mitigate | ADR-5: all I/O routes through api.ts (`/api/config*`, `/api/configarr/*`); enforced by the acceptance-criteria grep gate (`grep -c "fetch(\|/api/v3\|http://"` returns 0) in Task 2. |
| T-26-07 | Information disclosure | api_key shown in configarr form | accept | api_key is a `!secret`/`!env` literal; the Phase 25 backend never resolves it (Phase 25 SC#4), so only the tag literal is displayed, read-only. Acceptable on a LAN-trusted local tool. |
| T-26-08 | Tampering | readOnly UI bypass on api_key / quality_definition / media_naming | accept | readOnly is a UI affordance only; authority lives in the Phase 25 backend (D-04 api_key opaque, D-09 guard). No privilege boundary crossed by a single operator on a local tool. Low risk. |

No HIGH-severity threats. Block gate (high) not triggered.
