# Phase 28: Generate foundation - Context

**Gathered:** 2026-05-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Phase 28 builds the **mechanism** of the v0.10.0 intention layer — not its block logic.
It delivers:
1. `intent.yml` — a new hand-edited source file with the tranche-1 schema.
2. `arrconf generate` — a pure `intent → verbose configs` CLI subcommand whose output is **committed**.
3. A CI idempotence guard that fails on drift between intention and committed configs.
4. A new ADR formalizing the intention layer + the *absorber vs déployer* boundary (extends ADR-5).

**Out of this phase:** the block reconcilers/deployers (sagas → Phase 29, cross-seed Helm alias → Phase 30, qbit_manage CronJob → Phase 31), the UI-on-intention, configarr.yml generation, and migrating `categories[]` into `intent.yml` (INTENT-CATMIG-01, v2-deferred). P28 proves the rails; later phases ride them.

</domain>

<decisions>
## Implementation Decisions

### Bootstrap emission — what `generate` actually writes in P28
- **D-01:** "New blocks only" boundary. P28 wires `generate` end-to-end but emits ONLY the new intention-layer target files. `arrconf.yml`/`configarr.yml` stay hand-edited and untouched in tranche 1 (the in-memory `categories[]` expansion at apply-time is unchanged). The deferred categories→generated migration stays genuinely deferred.
- **D-02:** Proof-of-wiring via **seeded sample intent**. Ship a real `intent.yml` carrying one sample block entry that generates a committed, **non-empty** config file, so the CI guard has real bytes to diff. The block's *deploy* (Helm alias / CronJob) still waits for P30/31 — P28 proves `generate` only, not deploy.
- **D-03:** The proving slice is **cross-seed (`config.js`, JS literal)**, NOT qbit_manage YAML. This forces the generate framework to support literal-JS emission (`module.exports = {...}`) — the genuinely novel output format — right in the foundation, de-risking P30 and proving the framework is not YAML-only.

### intent.yml shape
- **D-04:** **Separate file** `charts/arr-stack/files/intent.yml`, hand-edited, sibling to `arrconf.yml`/`configarr.yml`. Tranche 1: `intent.yml` holds ONLY the new blocks. `arrconf.yml` keeps `categories[]` + `sonarr.main` etc. hand-edited as today. No categories migration.
- **D-05:** Top-level layout = `tools:` mapping + `sagas:` list, mirroring DESIGN §3 verbatim. P28 ships the schema for both; only `tools.cross_seed` is exercised. `tools.qbit_manage` and `sagas[]` schemas are present-but-unexercised until P31/P29 respectively (later phases fill behavior, not the root schema shape).

### generate CLI + apply relationship
- **D-06:** `generate` and `apply` are **fully decoupled**. Operator workflow: edit `intent.yml` → `arrconf generate` (writes files) → review diff → commit → `arrconf apply` reconciles the committed files. `apply` NEVER auto-runs `generate` (upholds G1: committed configs are the source `apply` reads; ADR-6 diff-in-PR discipline).
- **D-07:** `generate` exposes a **`--check` flag**: writes nothing, exits non-zero (and prints the diff) when output would differ from committed files. CI calls `arrconf generate --check`; operators run bare `arrconf generate` locally to write. Single code path. (This is the chosen realization of INTENT-03's `generate && git diff --exit-code` wording — satisfies the requirement with a cleaner, file-system-clean CI invocation.)
- **D-08:** `generate` follows the existing subcommand pattern (`apply`/`dump`/`diff`/`schema-gen` in `__main__.py`). Naming/flags beyond `--check` are planner discretion.

### CI idempotence guard mechanics
- **D-09:** Guard lives as a new job in **`tests.yml`** (runs on PR + push), NOT in `chart-lint.yml`. Rationale: `chart-lint.yml` carries the `mathieudutour/github-tag-action` auto-tagger that fires on push to main; keeping the guard out of that workflow isolates it from release-tagging order/interaction.
- **D-10:** Output config files (cross-seed `config.js` now; qbit_manage `config.yml` later) are written & committed **under `charts/arr-stack/files/`** — co-located with `arrconf.yml`/`configarr.yml` and ready for ConfigMap mounting by the P30/31 Helm aliases. Editing these never triggers the `arrconf-image.yml` rebuild (that watches `tools/arrconf/**` only). Exact sub-paths/filenames are planner discretion (see Deferred / planner-flag below re: generic `config.yml` filename — ConfigMap key/subPath can rename).

### Read-only marking (SC#1)
- **D-11:** Every generated file gets a header comment block: `# GENERATED by 'arrconf generate' from intent.yml — DO NOT EDIT BY HAND` (JS-comment equivalent `//` for `config.js`). Applied to **generated targets only** (cross-seed `config.js` in P28). `arrconf.yml`/`configarr.yml` stay hand-edited in tranche 1 and are NOT marked read-only yet — they are not generated until a future tranche.

### ADR (INTENT-04)
- **D-12:** Write one new ADR in `.planning/` (next free ADR number — verify highest existing before assigning). It formalizes: (a) the intention layer sitting **above** both arrconf and configarr; (b) the **absorber (generate the config) vs déployer-seulement (DB/UI-only)** boundary; (c) the **ADR-5 extension** — configarr remains the sole TRaSH applier, the intention layer does not touch quality_profiles/custom_formats/quality_definitions/media_naming. Content sourced from DESIGN §2 + §5.

### Claude's Discretion
- Subcommand flag surface beyond `--check` (D-08).
- Exact output sub-paths/filenames under `charts/arr-stack/files/` (D-10).
- Placement of the pydantic `IntentConfig` model (new module vs extend `config.py`) — must reuse the `arrconf/generators/` pure-function pattern per INTENT-02.
- Generate determinism/ordering details (deterministic output is mandatory for idempotence; mechanism is planner's).

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `arrconf/generators/categories.py`: the exact pure-function generator idiom to extend — `generate_X(cfg) -> typed resources`, no httpx/I/O, deterministic ordering follows input order. New `generate_cross_seed(intent) -> str` (JS literal) follows this shape but returns a rendered string rather than pydantic resources.
- `arrconf/config.py` pydantic models (`extra="forbid"`): `IntentConfig` schema reuses these conventions; `schema-gen` subcommand already exists to regenerate JSON Schema.
- `__main__.py` subcommand pattern: `generate` slots in beside `apply`/`dump`/`diff`/`schema-gen`.

### Established Patterns
- G1 (generate local → commit → apply reads committed): generators currently run in-memory at apply-time. P28 introduces the **file-output** variant for the new blocks WITHOUT converting the existing categories path (D-01).
- ruyaml round-trip for YAML; for `config.js` the output is a hand-rendered JS literal (no YAML lib) — the novel bit P28 proves (D-03).
- Co-bump pin discipline: `generate` code lives under `tools/arrconf/**` → image tag co-bump applies; the seeded `intent.yml` + generated `config.js` live under `charts/` → no image rebuild.

### Integration Points
- New `intent.yml` at `charts/arr-stack/files/intent.yml`.
- Generated outputs under `charts/arr-stack/files/` (ConfigMap-mount-ready for P30/31).
- New CI job in `tests.yml`.
- New ADR in `.planning/`.

</code_context>

<specifics>
## Specific Ideas

- Prove the hard output format first: cross-seed `config.js` `module.exports = {...}` literal emission is the deliberate P28 proving slice (D-03) — "if generate can emit valid JS, it can emit anything."
- The CI guard must have real bytes to diff — seeded sample intent, not empty skeletons (D-02).
- Read-only header is operator-facing guardrail copy: `DO NOT EDIT BY HAND` (D-11).

</specifics>
